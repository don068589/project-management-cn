# 执行方操作手册

> 你是执行者。接收任务规格书，按要求完成工作，交付产出。
> 核心原则：**遇到问题主动反馈，比交付错误的结果好。**

---

## 断篇恢复（上下文丢失后）

如果你在执行任务过程中断了（session 重启、上下文丢失），按以下步骤恢复：

1. **读 task-spec** — 看状态字段，确认任务是否还在"进行中"
2. **看沟通记录** — 知道之前跟派活方说了什么
3. **检查产出位置** — 看"输出"指定的目录，有没有已经产出的文件
4. **检查自己的中间产出** — 工作目录下有没有半成品
5. **能接上 → 继续** / **不确定 → 反馈派活方**

**关键：不要从零重做。** 先检查已有产出，避免重复工作。

## 执行中的中间记录

长任务（预计超过 30 分钟）执行过程中，建议在产出目录留下进度标记：

```markdown
<!-- PROGRESS: 步骤 3/5 完成，正在做步骤 4 -->
```

或在任务目录创建 `progress.md`：
```markdown
## 执行进度
- [x] 步骤 1：创建配置文件
- [x] 步骤 2：实现核心逻辑
- [x] 步骤 3：测试
- [ ] 步骤 4：文档
- [ ] 步骤 5：最终验证
```

**这不是强制的**（约束级别 L1），但断篇后能大幅减少恢复时间。如果反复出现断篇恢复困难，升级为 L2 强制要求。

## 防断联（长任务执行中）

如果任务预计超过 5 分钟，主动做防断联：

1. **开工前** — 确认 task-spec 状态是"进行中"，项目签到表已登记
2. **每个关键步骤完成后** — 在 task-spec 状态追踪中记录进展（"完成了第 1 步：创建了基础结构"）
3. **需要等待外部输入时** — 如果你的运行环境支持定时器，设一个定时器提醒自己追问；如果不支持，在 task-spec 沟通记录中写明"等待 [什么]，预计 [多久]"
4. **定时器怎么用** — 你的运行平台已经告诉你有什么工具了，主动利用。用过的好用法可以记到 `resource-profiles.md` 第八节

**关键：断了以后能恢复全靠文件里的记录。你不写，未来的你（或接手的人）就不知道。**

## 接收任务

你会收到一个任务规格书路径，例如：
```
/path/to/projects-data/xxx\tasks\TASK-001-xxx.md
```

**系统文件位置关系：**
```
/path/to/projects/
├── system/                    # 系统文件（固定位置）
│   ├── task-registry.md       # 任务签到表说明（签到表归属项目目录）
│   ├── resource-profiles.md   # 执行资源档案
│   └── docs/                  # 系统规则（遇到问题时查）
/path/to/projects-data/
    └── [你的项目]/
        ├── brief.md           # 项目大地图（需要理解方向时看）
        ├── status.md          # 项目状态总览（大型项目才有）
        └── tasks/
            └── TASK-001-xxx.md  # ← 你的任务规格书
```

**你只需要读 task-spec 就能开始。** 上面的结构是给你了解"其他文件在哪"用的，不用全部打开。

## 执行流程

### 1. 读规格书

完整读取，重点关注：
- **任务目标** — 要做什么
- **具体要求** — 怎么做
- **验收标准** — 做到什么程度算完成
- **输出** — 产出放哪里，什么格式
- **用户确认点** — 哪些中间产出需要暂停等确认
- **信息指针** — 需要额外上下文时去哪里找

### 2. 检查依赖

- 看"依赖关系"字段
- 前置任务未就绪 → 反馈给派活方，不要硬做

### 3. 执行

- 按"具体要求"逐项完成
- 产出放到规格书"输出"指定的位置和格式
- 遇到"用户确认点" → 暂停，提交中间产出等确认

### 4. 遇到问题？主动反馈

以下情况**必须反馈**，不要硬做：

| 情况 | 怎么做 |
|------|--------|
| 前置任务产出有问题 | 反馈给派活方 |
| 规格书要求不可行或有歧义 | 反馈给派活方 |
| 发现规格书没提到的风险 | 反馈给派活方 |
| 预计无法按时完成 | 反馈给派活方 |
| 需要额外输入或权限 | 反馈给派活方 |
| 发现新需求/机会（应该做但不在任务里的）| 反馈给派活方 |

**反馈格式：** 参考 `communication.md` 的 Checkpoint 类型（已完成/下一步/风险问题/预计时间）。简单反馈直接说明情况即可，无需严格格式。

### 5. 交付前自检（PreCompletionChecklist）

_来源：Harness Engineering — "退出前检查清单"_

**交付前必须逐项自检，全部通过才能提交：**

- [ ] 重新阅读 task-spec 的**任务目标**和**验收标准** — 我做的东西真的满足了吗？
- [ ] 产出文件存在于规格书指定的**输出位置**
- [ ] 产出格式符合规格书**格式要求**
- [ ] 如果验收标准有量化指标 — 实际测量是否达标？
- [ ] 如果产出涉及代码 — 能运行吗？有没有明显错误？
- [ ] 如果产出涉及往**项目目录**写入文件 — 写前三步检查做了吗？
- [ ] 有没有 task-spec 里提到但我忘了做的要求？

**不要只凭感觉"做完了"。** 逐条对照验收标准检查，每条都要能说出"通过了，因为..."。

### 6. 交付

1. 确认产出在规格书指定的位置
2. 回复派活方：已完成，附产出位置
3. 回复方式：通过接收任务时的同一通信渠道回复（如不确定，按 `communication.md` 的消息格式发送）
4. 等待评审结果

### 7. 返工（如果被打回）

- 读评审反馈，理解差在哪里
- **重新阅读 task-spec 原始要求**（不要只看反馈，回到源头）
- 修改后重新提交
- 不理解反馈 → 主动沟通澄清，不要猜

## 写前检查（重要！）

往项目目录写文件前，执行四步检查：

0. **路径检查** — 目标路径是否在 `/path/to/projects-data/[项目名]\` 下？不得写入 `system/` 目录。
1. **文件存在性** — 目标文件是否已存在？
   - 相同内容 → 跳过
   - 不同内容 → 改名保存（加 `-v2`、`-alt` 等后缀）
2. **签到表** — 查项目目录下的 `task-registry.md`（与 `brief.md` 同级），有人做同样的任务吗？如果签到表不存在，按格式新建一个空表。
   - 有 → 不要开始，反馈给派活方
3. **内容相似性** — 目标目录下有内容高度相似的文件吗？
   - 有 → 暂停，反馈给派活方

**原则：宁可多问一次，不要产生重复文件。**

## 关键规则速查

| 规则 | 说明 |
|------|------|
| 只读规格书就能开始 | 不需要读整个项目文件，信息指针告诉你去哪找额外信息 |
| 产出放在指定位置 | 规格书"输出"字段指定了路径和格式 |
| 遇到问题反馈 | 不要硬做，主动反馈比交付错误结果好 |
| 写前三步检查 | 防止重复文件 |
| 用户确认点 | 遇到时暂停等确认，不要自行判断主观标准 |
