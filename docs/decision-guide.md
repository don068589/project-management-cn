# 决策与编写指南

> 派活方在过程中需要做各种判断：什么时候打回、什么时候换人、怎么写反馈、怎么调整大地图。
> 本文件是"怎么做决定"和"怎么写文件"的元引导。

---

## 一、评审决策指南

### 什么时候通过

- 验收标准逐项满足
- 产出文件在指定位置
- 格式符合要求
- 无明显遗漏

### 什么时候打回

- 验收标准有未满足项
- 产出缺失或位置错误
- 质量明显低于预期

**打回时怎么写反馈：**
```markdown
## 评审反馈

**未通过项：**
1. [具体哪条验收标准] — [差在哪里] — [期望是什么]
2. ...

**已通过项：**（让执行者知道哪些不用改）
1. ...

**修改指令：**
- [明确告诉执行者要做什么修改]
```

**原则：反馈必须具体到"差在哪里 + 期望是什么"。不说"不够好"，说"缺少错误处理，需要加 try-catch 覆盖 X 和 Y 两种异常"。**

### 什么时候换人/换方案

| 情况 | 判断标准 | 动作 |
|------|----------|------|
| 返工 3 次仍不达标 | 问题是否在执行者能力范围内 | 换执行者或换方案 |
| 执行者反馈"不可行" | 评估反馈是否合理 | 合理→换方案，不合理→补充说明 |
| 执行者超时无响应 | 等待时间超过过期时间 | 重新分发 |
| 产出方向完全错误 | 执行者是否理解了任务 | 澄清→重做，理解力不足→换人 |

### 什么时候升级给决策者

- 自己判断不了的主观问题
- 涉及项目目标或范围的变更
- 成本或时间超出预期
- 返工 3 次仍无进展
- 多个方案各有利弊，需要拍板

## 二、文件编写指南

### 怎么写 task-spec

**好的 task-spec 标准：执行者只读这一份就能开始工作。**

| 要素 | 好 | 坏 |
|------|---|---|
| 任务目标 | "实现 GitHub Actions 自动打 tag" | "做好 CI" |
| 验收标准 | "push 到 main 后自动创建 vX.Y.Z tag" | "能正常工作" |
| 输出位置 | "`.github/workflows/release.yml`" | "放到合适的位置" |
| 注意事项 | "不要用 deprecated 的 actions/create-release" | _(空)_ |

**检查清单（写完后自查）：**
- [ ] 执行者不需要问我就能开始吗？
- [ ] 验收标准是可以逐条打勾的吗？
- [ ] 输出位置和格式明确吗？
- [ ] 信息指针只给了需要的上下文吗？（不要塞无关信息）

### 怎么写评审记录

- 逐条对照验收标准
- 每条标注 ✅ 通过 / ❌ 未通过
- 未通过的写明差距和期望
- 整体结论：通过 / 返工 / 升级
- 如果是返工，写明具体修改指令

### 怎么写变更请求

- 原始需求是什么（引用 brief.md 或 task-spec）
- 变更什么
- 为什么变更
- 影响分析：哪些任务受影响、需要多少额外工作
- 建议方案

### 怎么写 brief 调整（大地图变更）

大地图调整 = brief.md 需求确认记录新增一条：

```markdown
| 轮次 | 时间 | 调整内容 | 原因 | 决策者确认 |
|------|------|----------|------|-----------|
| 3 | YYYY-MM-DD | 范围调整：去掉某功能 | 时间不够 | 待确认 |
```

同时检查：
- 哪些已有 task-spec 受影响？→ 更新或取消
- 哪些新任务需要创建？→ 新建 task-spec
- 风险是否变化？→ 更新 risk-register

## 三、运行时内容编写指南

项目执行过程中会产生各种实时内容，不是从模板复制的，而是当场写的。怎么写：

### 怎么写 Heartbeat 检查项

```markdown
## [项目名]
- [当前最紧迫的待办] ← 只写下一步，不写历史
- [如果有第二个待办]
```

**原则：**
- 每个项目最多 3 条检查项
- 只写"下一步做什么"，不写已完成的事
- 项目完成 → 删除整个条目
- 每完成一个阶段立即更新

### 怎么写沟通记录条目

```markdown
| 时间 | 方向 | 内容摘要 | 通信标识 / 工具 |
|------|------|----------|-------------------|
| 03-21 10:00 | 派活方→执行者 | [一句话概括：做了什么/说了什么] | [标识符] |
```

**原则：** 摘要一句话够用。目的是恢复时知道"跟谁聊过什么"，不是记全文。

### 怎么写状态追踪条目

```markdown
| 时间 | 状态变更 | 说明 |
|------|----------|------|
| 03-21 10:00 | 待分发→进行中 | [原因/触发事件] |
```

### 怎么写公共留言板条目

```markdown
| 时间 | 留言人 | 内容 |
|------|--------|------|
| 03-21 10:00 | [角色/任务ID] | [信息内容] |
```

**何时用留言板 vs 沟通记录：**
- 只跟一个执行者相关 → 沟通记录
- 需要多人知道 → 留言板

## 四、新文件的记录与链接

**任何时候在项目目录中新建文件，必须完成两步：**

1. **在对应的上级文档中添加引用**
   - 新建 task-spec → 确认 brief.md 的任务列表提到了它
   - 新建 review-record → 确认 task-spec 的状态追踪引用了它
   - 新建 change-request → 确认 brief.md 或受影响的 task-spec 引用了它

2. **检查索引一致性**
   - 如果是系统级文件（system/ 下）→ 更新 data_structure.md
   - 如果是项目文件（/path/to/projects-data/ 下）→ 不需要更新系统索引，项目内部自链接

**原则：每个文件必须被至少一个其他文件引用，否则就是孤岛。**
