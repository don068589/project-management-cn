# 系统治理机制

> 约束升级、规则沉淀、模板版本、系统复盘、沙盒环境、Heartbeat 自循环。

---

## 约束三级火箭

有些规则写在文档里但没人遵守，同一问题反复出现。

**三个级别：**

| 级别 | 形式 | 执行方式 | 示例 |
|------|------|----------|------|
| L1 建议 | 人格/工作文件里的文字 | 靠执行者自觉 | "尽量量化验收标准" |
| L2 文档 | 系统流程规则 / skill 提醒 | 加载时注入提醒 | "评审前必须有评审记录" |
| L3 代码 | `tools/` 下的检查脚本 | 机械执行，自动拦截 | dashboard.py 检查状态不一致 |

**升级规则：**
- 同一问题第一次 → 口头提醒（L1）
- 再次出现 → 写进流程规则（L2）
- 第三次 → 写检查脚本自动拦截（L3）

**原则：当你在评审中反复纠正同一个问题，不要修任务，修规则。**

## 系统变更审批

**`system/` 目录包含两类文件，权限不同：**

### 系统规则文件（修改需审批）

定义系统"怎么运作"的文件。改了就是改规则。

- `docs/` 下所有文件（流程、原则、操作手册）
- `templates/` 下所有模板
- `README.md`（系统入口和变更日志）
- `data_structure.md`（系统索引）
- `tools/` 下的脚本

**变更流程：**
1. 发现需要改进 → 写入 `improvements.md`（待改进清单，系统数据文件，正常读写。含问题、建议、影响分析、来源）
2. 决策者充分听取分析和验证后决定：
   - ✅ 批准 → 执行修改 + 记录变更日志 + 提交 Git 快照（`python tools/system-check.py --commit "修改说明"`）
   - ❌ 否决 → 记录原因，关闭
   - 🔄 需要更多信息 → 补充后重新审阅
3. 唯一例外：明显的文字错误（typo、格式）可直接修复，但需记录

### 系统数据文件（正常读写）

记录运行时信息的"外部插口"。系统运行过程中自然产生和更新的数据。

**预置数据文件：**
- `resource-profiles.md` — 执行者名片、合作记录、积累规则
- `task-registry.md` — 任务签到表说明（签到表归属项目目录）
- `improvements.md` — 待改进清单（系统改进建议收集）

**可扩展性：** 数据文件不限于上述两个。派活方可以按需在 `system/` 下创建新的数据文件来记录新主题的运行时信息。例如：
- 记录常用环境配置 → 创建 `environments.md`
- 记录跨项目的技术栈偏好 → 创建 `tech-preferences.md`
- 记录常见错误和解决方案 → 创建 `troubleshooting.md`

**创建规则：**
1. 文件头部标注 `> 系统数据文件 — 正常读写，无需审批` 以区分
2. 更新 `data_structure.md` 索引
3. 在变更日志简要记录（不需要审批，但要留痕）

**可重建性：** 所有数据文件都是"经验积累"而非"系统骨架"。即使全部删除，系统规则文件完整无损，系统照常运作。下次运行时会自然重新创建空的数据文件：
- 第一次分发任务 → 发现项目签到表不存在 → 按格式新建
- 第一次跟新执行者合作 → 发现名片不存在 → 按模板新建
- 丢失的只是历史经验，不是运作能力

**这就是"外部插口"的本质：可插拔、可扩展、可重建。**

## 系统规则沉淀（增强版）

项目结束后的知识沉淀，增加三个系统级检查项：

1. **可复用规则 → 升级到系统**
   - 项目中发现的好规则，值得变成系统级流程？→ 写新流程机制
2. **反复问题 → 升级约束级别**
   - 当前约束级别够吗？→ 按约束三级火箭升级
3. **新任务类型 → 更新适配表**
   - 涉及了 `quality.md` 任务类型适配表中没有的类型？→ 补充

**系统进化链条：**
```
项目实践 → 发现问题/好规则 → 沉淀到系统 → 下个项目受益 → 持续改进
```

## 系统扩展流程

当需要给系统添加新功能（新模块、新模板、新工具）时，按以下流程操作：

### 添加新的 docs 模块

1. 在 `docs/` 下创建新文件，命名为 `主题名.md`
2. 文件头部写明定位和用途（一句话）
3. 更新 README.md 的索引表（第四节"需要查详细规则"）
4. 更新 data_structure.md 的目录结构
5. 更新 self-loop.md 的"时机→文件"速查表（添加"什么时候读这个文件"）
6. 更新 self-loop.md 的引导文件层次结构图
7. 在 README.md 变更日志中记录

### 添加新的模板

1. 在 `templates/` 下创建新模板
2. 更新 README.md 的模板清单（第五节）
3. 更新 data_structure.md
4. 更新 coordinator.md 或 executor.md 中引用模板的地方（如果有）
5. 在 README.md 变更日志中记录

### 添加新的工具脚本

1. 在 `tools/` 下创建脚本
2. 更新 data_structure.md
3. 在对应的 docs 文件中说明用途和用法
4. 在 README.md 变更日志中记录

### 修改现有模块

1. 直接修改对应的 docs 文件
2. 检查其他文件中是否有引用该模块的内容 → 如需则同步更新
3. 在 README.md 变更日志中记录

### 检查清单（扩展完成后自查）

- [ ] 新文件有没有被至少一个其他文件引用？
- [ ] README.md 索引表是否更新？
- [ ] data_structure.md 目录结构是否更新？
- [ ] self-loop.md 速查表是否更新？
- [ ] 变更日志是否记录？
- [ ] 对进行中项目有没有影响？有 → 通知相关执行者（用 communication.md Share 类型）

### 系统升级与进行中项目

- **新增机制（加法）** → 一般不影响进行中项目，新项目自动使用
- **修改现有机制** → 检查是否影响正在执行的任务
  - 不影响 → 静默升级
  - 影响 → 通知相关的派活方和执行者，说明变更内容
- **模板变更** → 只对变更后创建的任务生效，旧任务不受影响

## 模板版本兼容

- 模板变更不影响已发出的任务 — 按创建时的版本检查
- 新字段只对变更后创建的任务生效
- 重大变更在变更日志中标注"不向后兼容"
- 旧任务需适配新模板 → 视为变更请求处理

## 系统复盘

每完成 3-5 个项目，或每月一次（取先到者），进行系统复盘：

- 流程是否顺畅？哪里卡过？
- 模板是否够用？哪些字段没人填？
- 评审质量如何？返工率高不高？
- 资源分配是否合理？
- improvements.md 待改进清单是否积压？

复盘结果更新到 `improvements.md` 的待改进清单和 README.md 的变更日志。

## 垃圾回收机制（对抗熵增）

_来源：OpenAI Harness Engineering — "周期性运行的 agent，发现文档不一致或架构约束违规"_

项目运行久了，状态会腐化：签到表有过期条目、task-spec 状态和实际不符、索引遗漏、名片过时。

**垃圾回收 = 周期性健康扫描。** 不是等出了问题才修，而是定期主动发现问题。

### 扫描清单

| 检查项 | 检查方式 | 频率 |
|--------|---------|------|
| 项目签到表有无过期未清理的条目 | 各项目 task-registry.md 中"进行中"超过 7 天的条目 | 每次系统级 Heartbeat |
| task-spec 状态与项目签到表是否一致 | 交叉对比 | 每次系统级 Heartbeat + 每次冷启动恢复时抽查 |
| data_structure.md 索引与实际文件是否一致 | 扫描目录 vs 索引 | 每月 |
| 模板文件是否完整（未被意外删除/损坏）| 检查 templates/ 下文件数和文件头 | 每月 |
| system/ 规则文件完整性 | `python tools/system-check.py`（基于 Git 对比） | 每次系统级 Heartbeat |
| 名片中"最近合作时间"超过 3 个月的 | 标记为"待更新"或"待清理" | 每月 |
| 文档交叉引用有效性 | docs/ 内 `引用.md` 指向的文件是否存在、节标题是否匹配 | 每月 |
| /path/to/projects-data/ 下已完成项目是否归档 | 检查 status 为已完成但仍在 /path/to/projects-data/ 的 | 每月 |
| Heartbeat 中是否有已完成项目的残留条目 | 读 Heartbeat 对比项目状态 | 每次系统级 Heartbeat |

### 执行方式

- **L1（起步）：** 写入系统级 Heartbeat，派活方每次心跳手动检查
- **L2（成熟）：** 用 dashboard.py 或类似脚本自动扫描，输出报告
- **L3（完善）：** 自动修复简单问题（如清理过期签到表条目），复杂问题生成修复建议

### 扫描结果处理

- 发现不一致 → 修复（数据文件直接修、规则文件走审批）
- 发现系统性问题（同一类不一致反复出现）→ 升级约束级别
- 记录到 `/path/to/projects-data/lessons\garbage-collection-log.md`（系统数据文件，正常写）

## 沙盒环境利用

沙盒是可用的执行环境，根据任务性质灵活使用：

| 场景 | 说明 |
|------|------|
| 在沙盒中制作 | 开发、测试、实验性任务，完成后搬出产出 |
| 沙盒作为交付物 | Web 应用、服务等，沙盒本身是运行环境 |
| 沙盒验证 | 先在沙盒验证可行，再正式执行 |

需要访问本地文件系统的任务必须在主机执行。

## Heartbeat 自循环机制

_注：适用于支持 heartbeat 机制的 agent 框架。不支持 heartbeat 的环境，由派活方手动定期检查。_

### 项目级 Heartbeat（高频）

项目启动后，总指挥在 heartbeat 配置中写入项目检查任务，利用心跳自动续接。

**流程：**
1. 项目启动 → 写入项目检查条目
2. 每次心跳 → 检查项目状态
3. 有待办 → 自动推进（分发、评审、催进度）
4. 项目完成 → 删除检查条目

**优点：**
- 不依赖持久 session，断了自动续接
- 项目文件就是记忆
- 完成后自动清理

**重要：随项目进展更新检查内容，不要写死不动。**

### 系统级 Heartbeat（低频）

除了项目检查，Heartbeat 中还应包含系统层面的周期性检查：

```markdown
## 系统维护（每月检查一次）
- 上次系统复盘：[日期]（超过 1 个月？→ 做复盘，流程见本文件"系统复盘"节）
- improvements.md 待改进清单积压了吗？
- 有没有约束需要从 L1 升级到 L2/L3？（回顾最近项目的评审反馈）
- resource-profiles.md 名片是否需要更新？
- 垃圾回收扫描（见本文件"垃圾回收机制"节的扫描清单）
```

**触发条件：** 每完成 3-5 个项目，或上次复盘超过 1 个月。
**复盘流程见本文件"系统复盘"节。**

### 两层 Heartbeat 的写法

```markdown
## [项目名 A]（高优先级）
- TASK-003 进行中，检查是否完成
- TASK-004 待分发，依赖 TASK-003

## [项目名 B]（中优先级）
- 所有任务待分发，优先 TASK-001

## 系统维护
- 上次复盘：YYYY-MM-DD（下次：YYYY-MM-DD）
```
