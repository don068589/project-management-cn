# 调度与资源管理

> 资源分配、优先级、异常处理、紧急任务、能力画像。

---

## 动态资源分配

**核心原则：角色不固定，资源动态分配。**

每次分发任务时，根据当下可用资源决定谁来做，不预设固定分工。

**选择依据：**
- 任务复杂度和独立性
- 当前可用的通信通道
- 执行者的能力匹配度（参考 `resource-profiles.md`）
- 成本效益（token 消耗、时间）

## 优先级调度

- 项目级优先级在 brief.md 标注（高/中/低）
- 任务级优先级在 task-spec 标注（高/中/低）
- 同优先级按依赖关系排序
- 资源冲突时，高优先级优先

| 情况 | 处理方式 |
|------|----------|
| 高优先级到来 | 暂停低优先级任务 |
| 多个高优先级并行 | 无依赖的并行分发 |
| 资源不足 | 汇报决策者 |
| 紧急任务插入 | 立即响应，必要时暂停当前工作 |

## 异常处理与升级

| 异常 | 检测方式 | 处理 |
|------|----------|------|
| 执行超时 | 超过截止时间 | 催进度 → 无响应则重新分发 |
| 执行者失联 | 无响应 | 等待合理时间 → 重新分发或换执行者 |
| 任务卡住 | 执行者报告 | 分析原因 → 调整/换方案/升级 |
| 质量持续不达标 | 返工超 3 次 | 升级决策者 |
| 依赖阻塞 | 前置延期 | 评估能否并行 → 调整 → 汇报 |
| 资源不可用 | 工具故障 | 换执行方式 → 汇报 |

**升级路径：**
```
自主处理 → 汇报知悉 → 请决策者拍板 → 决策者直接介入
```

**原则：** 小问题自己解决，大问题及时升级。不要在死胡同反复尝试超过 3 次。

## 紧急任务快速通道

**触发条件：** 决策者明确表示紧急。

**规则：**
- 跳过立项，直接写 task-spec 并分发
- 跳过需求确认（紧急指令本身就是确认）
- 评审可简化但不跳过
- **事后补文档是强制的**（见下方检查清单）
- 质量门禁仍然有效

**事后补文档检查清单（紧急任务完成后 24 小时内）：**
- [ ] 补写 brief.md（如果跳过了立项）
- [ ] task-spec 验收标准是否完整？不完整则补上
- [ ] review-record 是否创建？简化评审也要有记录
- [ ] 沟通记录是否完整？
- [ ] data_structure.md 索引是否更新？
- [ ] 写入 Heartbeat 提醒"检查紧急任务 XXX 的补文档"

**谁来确保：** 派活方在紧急任务完成后，将"补文档"作为一个独立待办写入 Heartbeat。下次心跳时自动提醒。

**限制：** 快速通道不是常态。频繁使用说明流程有问题。

## 紧急任务与进行中任务的冲突

- 紧急任务优先级高于一切非紧急任务
- 需要同一执行者：评估进行中任务状态 — 快完成就等，刚开始就暂停
- 可以换执行者：优先换人，不打断进行中任务
- 紧急任务完成后，立即恢复暂停的任务

## 并行资源冲突检测

- 分发前检查目标执行者是否有进行中任务
- 有冲突：高优先级优先，同优先级先到先得
- 同一执行者最多 2 个进行中任务

## 资源能力画像

系统级能力画像在 `resource-profiles.md`，记录执行资源的强项、弱项和适合的任务类型。

**维护规则：**
- 初始画像基于已知信息建立
- 每次项目结束后根据实际表现更新
- 分配任务时参考，但不是硬约束
