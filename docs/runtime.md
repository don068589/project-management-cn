# 运行时自维持机制

> 确保 agent 在任何断点恢复后，能完整接上之前的工作。
> 核心原则：**不靠记忆，靠文件。每个环节结束时留下足够的线索让下一次的自己（或别人）接上。**

---

## 交班记录机制

每个工作环节结束时，必须在对应文件中留下"交班记录"，告诉下一次醒来的自己（或接手的人）：

1. **刚做完什么**
2. **当前状态是什么**
3. **下一步该做什么**
4. **有没有待处理的问题**

### 交班记录写在哪

不需要单独的文件。交班信息分布在已有文件的标准字段中：

| 环节 | 交班信息在哪 | 具体字段 |
|------|-------------|----------|
| 立项完成 | brief.md | 需求确认记录 → 最后一条记录状态 |
| 拆解完成 | 各 task-spec | 状态字段（全部为"待分发"）|
| 分发完成 | task-spec + 项目 task-registry.md | 状态改为"进行中" + 项目签到表登记 |
| 评审完成 | task-spec + review-record | 状态改为"已验收/返工" + 评审结论 |
| 返工指令 | task-spec 沟通记录 | 最后一条记录 = 反馈内容 |
| 项目完成 | final-report + data_structure.md | 验收报告 + 索引状态更新 |

### 冷启动恢复流程

**agent 醒来后的标准恢复步骤（无论是心跳触发、新对话、还是被派活）：**

**重要：每次心跳触发时，都要走冷启动恢复。** Heartbeat 告诉你"关注哪些项目"，冷启动恢复告诉你"项目实际状态是什么"。两者配合：先扫描文件恢复真实状态，再处理 Heartbeat 中的检查项。

```
1. 读 data_structure.md → 知道有哪些活跃项目
2. 进入项目目录：
   a. 有 status.md？→ 读 status.md 恢复全局状态（大型项目）
   b. 没有 status.md？→ 读 brief.md 状态 + 逐个扫描 task-spec 状态字段
   c. 检查有无"返工"状态的任务 → 需要重新分发
   d. 检查有无"待分发"且依赖已就绪的任务 → 可以分发
   e. 检查有无"待评审"的任务 → 需要评审
3. 有事做 → 做
4. 没事做 → 等待
```

**上下文大小：** data_structure.md (~20行) + 每个 task-spec 只看前 5 行的状态 = **~50行即可恢复全局状态**

### Heartbeat 检查项的动态更新

Heartbeat 不是写死一份不动的。每完成一个阶段，更新检查项：

**阶段 1 — 拆解完成：**
```markdown
## auto-release 项目
- 三个任务待分发（TASK-001/002/003）
- TASK-001 无依赖，优先分发
```

**阶段 2 — TASK-001 已分发：**
```markdown
## auto-release 项目
- TASK-001 进行中（通信标识: xxx），检查是否完成
- TASK-002/003 等待前置
```

**阶段 3 — TASK-001 待评审：**
```markdown
## auto-release 项目
- TASK-001 待评审，需要评审
- TASK-002 前置就绪，评审通过后可分发
```

**原则：Heartbeat 里只写"下一步做什么"，不写历史。历史在项目文件里。**

**多项目时的优先级：**
- 按项目优先级排序（高→中→低）
- 同优先级内按紧迫度排序（有人等 > 有截止时间 > 常规）
- 紧急项目永远排最前面
- 每个项目最多 3 条检查项
- 系统维护条目放在最后（低频，不影响项目推进）

**系统维护条目示例：**
```markdown
## 系统维护
- 上次复盘：[日期]（下次：[日期]）
```

详细的系统级 Heartbeat 规则见 `governance.md`。

## 中间事件记录

项目执行过程中会产生各种中间事件（评审打回、变更请求、紧急插入等）。每种事件都有明确的记录位置：

| 中间事件 | 记录在哪 | 格式 |
|----------|----------|------|
| 评审反馈（打回） | task-spec 沟通记录 + review-record | 时间、反馈内容、需改进项 |
| 给执行者的新指令 | task-spec 沟通记录 | 时间、指令内容、通信标识 |
| 需求变更 | changes/CR-XXX.md | change-request 模板 |
| 大地图调整 | brief.md 需求确认记录 | 新增一条确认记录，标注调整内容 |
| 风险变化 | risk-register.md | 更新风险条目 |
| 执行者反馈的问题 | task-spec 沟通记录 | 时间、问题描述、处理方式 |
| 换执行者 | task-spec 状态追踪 + 沟通记录 | 标注"换执行者"、原因、新 通信标识 |
| 任务放弃换方案 | task-spec 状态改为"已放弃" + 新 task-spec | 引用原任务编号 |

**原则：每个事件发生时就地记录，不要事后补。记录的位置就是对应文件的标准字段，不需要另建文件。**

## 索引更新检查点

索引（data_structure.md）必须与实际文件保持同步。以下时机必须检查并更新索引：

| 时机 | 检查什么 |
|------|----------|
| 新建项目 | 活跃项目表是否更新 |
| 项目完成/暂停/取消 | 项目是否从活跃表移走 |
| 新建系统文件（docs/、tools/） | 目录结构是否更新 |
| 每次 Heartbeat 冷启动 | 扫一眼索引与 /path/to/projects-data/ 目录是否一致 |

**自检方法：** `ls /path/to/projects-data/` 的目录列表 vs data_structure.md 的活跃项目表，两者应一致。

### 时间预算检查

**长时间任务（>30分钟）执行中：**

1. **开工时** — 在 task-spec 状态追踪写"开始执行，预计 XX 完成"
2. **过半时** — 如果执行时间超过预计 50%，自问：
   - 当前进度是否符合预期？
   - 是否需要调整预计时间？
   - 是否需要反馈派活方？
3. **超时时** — 如果执行时间超过预计 100%，必须：
   - 在沟通记录里写明原因
   - 通知派活方是否需要延期或调整

**派活方责任：**
- 分发时写明预计时间
- 超过预计 50% 没收到交付 → 主动追问
- 超过预计 100% → 判断是否需要调整或换方案

## 断点恢复的完整检查清单

**任何时候被打断又回来，按此清单恢复：**

- [ ] 读 data_structure.md — 有哪些活跃项目？
- [ ] 每个活跃项目：扫描 task-spec 状态 — 哪些在进行中/待评审/待分发？
- [ ] 有"进行中"的任务 → 检查沟通记录的 通信标识，能否联系执行者？
- [ ] 有"待评审"的任务 → 开始评审
- [ ] 有"待分发"且依赖就绪的 → 分发
- [ ] 有"返工"的任务 → 检查是否已重新提交
- [ ] 项目签到表 task-registry.md → 有无过期未清理的记录？与 task-spec 状态是否一致？
- [ ] Heartbeat 检查项 → 是否反映当前实际状态？（不反映就更新）

## 持续工作时的防断联机制

_本节解决的问题：agent 在长时间工作中随时可能断联（session 超时、上下文溢出、网络中断）。如果断了但没有留下足够线索，恢复时会浪费大量时间甚至重复工作。_

### 元认知：主动利用平台能力为自己服务

**你知道自己有什么工具。** 你的运行平台已经告诉你了 —— 定时器、Heartbeat、消息通道、文件系统等等。这里列举的只是常见的几种，**你的平台可能还有更多能力，不要局限于本文提到的这些。** 主动探索、主动利用。

**但"知道有"和"主动用"是两回事。**

这是一种元认知能力：**意识到自己随时可能断联，并主动利用已有的平台工具来保护工作连续性。**

具体来说，你需要养成三个习惯：

1. **为自己设保险** — 开始长任务前，主动设定时器或更新 Heartbeat，确保万一断了有人叫醒你
2. **为未来的自己写笔记** — 工作中定期在文件里记录进展，因为下次醒来的你不记得任何事
3. **主动选择休眠策略** — 需要等待时，评估等多久，选择合适的唤醒方式（定时器精确叫醒 vs Heartbeat 定期巡检）

**不要被动等待。** 你的平台给了你定时器、Heartbeat、文件系统这些工具。**主动编排自己的工作节奏。** 用过的好用法记到 `resource-profiles.md` 第八节。

### 三种保险工具

| 工具 | 特点 | 适用场景 |
|------|------|---------|
| **Heartbeat** | 框架层固定间隔触发，agent 可以编辑检查内容 | 常规轮询、项目推进、系统维护 |
| **定时器（Cron）** | agent 自己设定精确时间触发，一次性或重复 | "等 5 分钟后检查结果"、"明天 9 点提醒" |
| **中间记录** | 不叫醒，但在文件里留下线索让下次能续上 | 长任务执行中的进度保存 |

**Heartbeat 不能替代定时器。** Heartbeat 是"定期巡检"，定时器是"精确叫醒"。需要精确时间的场景必须用定时器。

**以上只是常见的三种。** 你的平台可能提供更多工具（事件监听、webhook、后台任务等）。不要局限于这张表，主动发现和利用。

### 防断联三步法（长任务执行中）

**agent 在执行超过 5 分钟的工作时，应该主动做防断联：**

#### 第一步：开工前设保险

刚开始一个可能持续较长时间的工作环节时：

1. **更新 Heartbeat** — 写明当前正在做什么、做到哪了、下一步是什么
2. **如果需要精确唤醒** — 设定时器（如"15 分钟后检查 TASK-003 是否有结果返回"）
3. **task-spec 状态已更新** — 确保即使现在断了，文件里的状态是对的

**定时器触发消息的写法：** 定时器触发时你可能已经忘了上下文。消息必须包含足够信息让未来的自己立即知道该做什么：

```
❌ "检查一下任务"
✅ "定时器提醒：检查 auto-release 项目 TASK-002（路径：/path/to/projects-data/auto-release\tasks\TASK-002-api-design.md）是否有执行者回复。如果有 → 评审。如果没有 → 通过沟通记录中的通信标识追问。"
```

**原则：定时器消息 = 一个微型 task-spec。** 包含：做什么、在哪里、判断标准、下一步动作。

#### 第二步：工作中定期保存

每完成一个关键步骤：

1. **更新 task-spec 状态追踪** — 记录进展
2. **更新 Heartbeat** — 把"下一步"更新为真正的下一步
3. **如果在等待外部结果** — 设定时器叫醒自己检查

**频率判断：**
- 任务预计 5-15 分钟 → 开头和结尾各保存一次
- 任务预计 15-60 分钟 → 每个关键步骤保存一次
- 任务预计超过 1 小时 → 按里程碑分段，每段结束保存

#### 第三步：等待时主动休眠

当需要等待外部结果时（执行者完成、API 返回、人类确认）：

1. **评估等待时间** — 大概多久？
2. **选择工具：**
   - 等待 < 5 分钟 → 不休眠，轮询或继续其他工作
   - 等待 5-30 分钟 → 设定时器，精确叫醒
   - 等待 30 分钟以上 → 写入 Heartbeat，等下次心跳检查
   - 等待不确定时长 → Heartbeat + 定时器双保险
3. **休眠前确保：**
   - Heartbeat 内容反映当前状态
   - task-spec 状态已更新
   - 项目签到表已更新

### Heartbeat 间隔自适应（决策指南）

不同项目阶段对检查频率的需求不同。如果框架支持调整 Heartbeat 间隔：

| 项目阶段 | 建议间隔 | 理由 |
|----------|---------|------|
| 刚分发任务，等待执行 | 15-30 分钟 | 需要及时发现完成或问题 |
| 所有任务进行中，无阻塞 | 30-60 分钟 | 常规巡检即可 |
| 有任务待评审 | 尽快 | 评审是流水线瓶颈 |
| 等待决策者确认 | 60 分钟+ | 人类响应时间不确定 |
| 无活跃项目 | 最长间隔 | 只需系统维护检查 |

**如果框架不支持动态调整间隔：** 用定时器补充。Heartbeat 保持固定间隔做常规巡检，紧急的用定时器精确叫醒。

### 所有角色都适用

**这套机制不只是派活方的。** 每个 agent 角色都应该：

1. **执行者** — 长任务中定期写中间记录（executor.md 已有建议）；等待输入时设定时器提醒自己追问
2. **派活方** — 等执行结果时设定时器检查；多项目并行时按优先级设不同频率的检查
3. **独立验证者** — 验证完立即通知，不需要防断联（一次性任务）
4. **任何 agent** — 被分配工作后第一件事：确保有"叫醒自己"的机制生效

### 与现有机制的关系

```
防断联三步法 → 解决"工作中突然断了怎么办"
    ↓
中间记录 + Heartbeat + 定时器 → 三种保险工具
    ↓
冷启动恢复 → 解决"断了之后回来怎么续上"
    ↓
断点恢复清单 → 具体续上的步骤
```

**防断联是"预防"，冷启动恢复是"治疗"。两者都不能少。**
