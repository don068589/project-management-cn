# 设计理念与元能力

> 本文件是整个项目管理系统的灵魂。记录设计的"为什么"和"怎么想的"。
> 任何人读了这个文件，就知道系统的设计脉络、核心原则、思维方式。
> 需要升级系统时，先读这个文件，确保新设计符合这些原则。

---

## 一、顶层设计原则

### 1. 平台无关

系统的所有方法论、流程、标准不依赖任何特定平台或工具。

- ❌ "用 sessions_spawn 派活" — 这是某个平台的 API
- ✅ "派活时给执行者完整的任务描述和验收标准" — 这是方法论

平台会变（今天是 OpenClaw，明天可能换别的），但"怎么拆任务、怎么评审、怎么沟通"这些底层方法不变。

**平台特定的实现细节怎么办？** 记录在 `resource-profiles.md` 的工具档案中。探索出了某个平台的快捷用法就记下来，下次不用重新探索。但如果记忆丢了，靠底层方法论也能重新摸索出来。

### 2. 文件即记忆

agent 没有持久记忆。一切持久的东西必须在文件里。

- 做了什么 → 写在 task-spec 状态里
- 决定了什么 → 写在 brief.md / change-request 里
- 跟谁聊了什么 → 写在沟通记录里
- 下一步做什么 → 写在 Heartbeat 里
- 学到了什么 → 写在合作备忘 / 经验库里

**不在文件里 = 不存在。** 没有"我记得上次..."这种事。

### 3. 上下文预算

agent 的上下文窗口是有限资源。系统设计的每个环节都要控制读取量。

- 派活方开工 → 读 coordinator.md（~4KB），不是读整个系统（~40KB）
- 执行方接活 → 读 executor.md（~1.5KB）+ task-spec（~2KB）
- 冷启动恢复 → 扫描 status.md 或状态字段（~50行），不是读所有文件

**检验标准：** 如果某个环节需要读取超过 5KB 的内容才能开始工作，说明信息架构有问题，需要重新设计。

**派活方的上下文管理：**
- 一次只做一个环节（不要同时评审多个任务）
- 评审时只打开当前 task-spec + 产出文件 + review-record 模板
- 不需要打开 brief.md（除非需要确认整体方向）
- 切换项目时"放下"上一个项目的上下文（一切都在文件里，不需要记）
- 大型项目用 status.md 做全局视图，不用打开所有 task-spec

### 4. 模块化可递进

每个模块独立成文件，改一个不影响其他。

- 要加新机制 → 创建新文件，更新索引（见 governance.md 系统扩展流程）
- 要改某个机制 → 只改那个文件
- 要升级整个系统 → 逐个模块升级，不用推翻重来

**检验标准：** 如果改了一个文件需要同时改 3 个以上其他文件，说明耦合太紧，需要解耦。

**拆分 vs 调取的平衡：**

拆分解决的是**存储和维护**问题 — 改一个模块不影响其他。
引用解决的是**使用**问题 — 需要时通过链路一次性加载多个模块。

两者不矛盾。文件分开存储，但在使用时通过"场景调取清单"（见 self-loop.md）一次性加载所有相关模块。

**拆分的判断标准：**
- 两块内容**总是一起被需要**，且合并后 ≤5KB → 合并
- 两块内容**有时只需其中一块** → 保持分开
- 文件 <500字节 → 考虑合并到上级（太碎片化）
- 文件 >8KB → 考虑拆分（太大，一次读完影响上下文预算）

**核心认知：智能体掌握的信息越多，处理问题能力越强。** 拆分不是为了让智能体少读，而是为了让它在每个场景读到**恰好需要的全部信息**——不多不少。

### 5. 冷启动友好

假设每次开始工作时，agent 完全不记得之前发生了什么。系统必须能从文件完整恢复。

- data_structure.md → 知道有哪些项目
- task-spec 状态字段 → 知道每个任务的进展
- 沟通记录 → 知道跟谁聊过什么
- Heartbeat → 知道下一步做什么

**即使所有对话历史都丢了，只要文件在，就能接上。**

### 6. 约束优先于建议

建议会被忘记，约束不会。

- L1：建议（文字提醒）— 可能被忽略
- L2：流程（写在操作手册的检查清单里）— 跟着走就不会漏
- L3：代码（脚本自动检查）— 机械执行，不可能忽略

同一个问题反复出现 → 升级约束级别。这就是约束三级火箭。

### 7. 事前预判优于事后恢复

不只是"出了问题怎么恢复"，更重要的是"开始前预判可能出什么问题"。

coordinator.md 的每个环节都有"开始前预判"检查项。不是走完了才检查，而是走之前就想一遍。

## 二、元能力

### 什么是元能力

元能力 = "做事情的能力的能力"。

- 写代码是能力 → "知道什么时候该写代码、什么时候不该"是元能力
- 跟 agent 沟通是能力 → "面对不熟悉的 agent 平台能探索出沟通方式"是元能力
- 管项目是能力 → "知道什么时候该加新机制、什么时候是过度设计"是元能力

### 元能力清单

#### 1. 探索未知系统的能力

面对一个没接触过的执行者/平台/工具，怎么上手？

```
步骤：
1. 它是什么类型？（有持久记忆的 agent / 一次性执行者 / CLI 工具 / 人类）
2. 它能接受什么格式的输入？（自然语言 / 结构化指令 / API 调用）
3. 它的输出是什么形式？（对话回复 / 文件产出 / 命令行输出）
4. 它理解项目管理术语吗？（理解 → 直接派活 / 不理解 → 需要翻译层）
5. 它有没有特殊限制？（上下文长度 / 超时 / 权限）
```

**探索后立即记录到 resource-profiles.md。** 下次不用再探索。

#### 2. 适配新平台的能力

当前系统运行在某个平台上（可能是 OpenClaw、可能是别的）。如果平台换了：

1. 底层方法论不变（本文件 + docs/ 下的所有机制）
2. 平台特定的实现方式需要重新探索
3. 探索后更新 resource-profiles.md
4. 如果新平台有更好的功能 → 记录到合作备忘

**系统的核心价值不在于"用什么工具"，而在于"怎么组织工作"。**

#### 3. 翻译能力

不同执行者说不同的"语言"：

- 有些理解"验收标准"、"用户确认点"这种术语 → 直接用项目管理语言
- 有些只理解具体指令（"在这个路径创建这个文件"）→ 需要翻译
- 翻译的模板和示例见 communication.md

**关键：翻译不是简化。** 是把同样严谨的要求用对方能理解的方式表达。

#### 4. 中间层设计能力

当执行者不理解项目管理系统时（如 CLI 编码工具），可以设置专人中间层：

```
派活方 → 中间层 → 执行者（CLI 工具）
   ↑         ↓
   └── 结果回传 ←─┘
```

中间层的职责：
- 接收 task-spec → 翻译成 CLI 能理解的指令
- 调用 CLI → 监督执行过程
- 检查产出 → 不达标则返工（在中间层内部循环，不打扰派活方）
- 产出达标 → 格式化结果回传给派活方

**什么时候用中间层：**
- 执行者不理解项目管理术语
- 需要反复调教才能出合格产出
- 派活方不需要关心执行细节

**什么时候不用（直接派活）：**
- 执行者能理解 task-spec
- 任务简单，一次就能出结果

#### 5. 自我诊断能力

系统运行过程中怎么发现问题？

- 重复执行同一任务 → 任务认领机制有漏洞
- 返工超过 3 轮 → task-spec 描述不清 或 执行者能力不匹配
- 冷启动后找不到进度 → 交班记录没写
- 两个执行者产出冲突 → 写前检查没做
- 上下文爆炸 → 信息披露层次有问题

**诊断后的动作：** 找到对应的 docs 文件，检查是否有规则覆盖。有 → 执行没到位，加强约束。没有 → 补规则，按系统扩展流程添加。

#### 6. 名片与档案能力

每个合作过的执行者都应该有一份"名片"（完整档案），记录在 `resource-profiles.md` 中。

**名片的生命周期：**

```
首次合作 → 创建名片（探索能力/边界/沟通偏好）
    ↓
每次合作后 → 追加合作记录 + 更新能力认知
    ↓
收到 Feedback → "以后记住"条目写入积累规则
    ↓
下次合作前 → 先读名片（不用重新探索）
    ↓
平台变了 → 只需重新探索"调用方式"一栏，其他不变
```

**名片包含：**
- 基本信息（类型、首次/最近合作时间）
- 能力与边界
- 沟通偏好（语言、格式、特殊注意）
- 当前平台下的调用方式（平台换了只改这一栏）
- 积累规则（从历次 Feedback 沉淀）
- 合作记录（每次合作的摘要和教训）

**名片的价值：** 
- 记忆丢了 → 读名片就能恢复合作模式
- 换人接手 → 读名片就知道怎么跟这个执行者合作
- 平台换了 → 能力/偏好不变，只重新探索调用方式

详细的名片模板和管理机制见 `resource-profiles.md`。

#### 7. 自主解决问题的能力

**本系统是引导，不是枷锁。你有大脑，用它。**

系统不可能覆盖所有场景。当你遇到系统没有明确规定的情况时：

1. **先查系统** — 看 docs/ 里有没有相关机制。可能有，只是你没找到。用 self-loop.md 速查表定位。
2. **没有对应规则？** — 这很正常。系统覆盖的是高频场景和关键约束，不是所有情况。
3. **自己思考解决** — 你有推理能力，有判断力。在不违背系统纪律的前提下，自主决策。
4. **主动利用可用工具** — 你知道自己的平台有什么工具（定时器、消息通道、文件系统等，包括但不限于这些 — 你的平台可能还有更多能力）。不要等别人告诉你用，主动用。本系统提到的工具只是引导，不是完整清单。
5. **事后记录** — 遇到了新场景并自主解决了？记录到经验库。如果这个场景可能反复出现，建议沉淀为系统规则。

**两条红线：**
- ✅ 可以自主决策的：执行方式、工具选择、时间安排、沟通方式、问题排查
- ❌ 不能自主突破的：系统规则文件（docs/、templates/、README.md）的修改需审批、任务状态转换必须满足守卫条件、项目签到表纪律、质量门禁

**简单说：在系统纪律内自由发挥。遇到系统没覆盖的，先想再做再记。**

## 三、系统升级指南

### 什么时候需要升级

- 发现了系统没覆盖的场景
- 同一个问题反复出现（约束需要升级）
- 引入了新类型的执行者
- 项目规模超出了当前机制的承受范围
- 外部有好的方法论值得借鉴

### 怎么升级

1. **先读本文件（philosophy.md）** — 确认新设计符合设计原则
2. **确定改哪个模块** — 查 self-loop.md 的"时机→文件"速查表
3. **按 governance.md 的系统扩展流程执行** — 创建文件、更新索引、更新速查表
4. **检验：** 改了之后，其他模块需要改吗？如果需要改 3 个以上，说明设计有问题

### 升级的原则

- **加法优于改法** — 能新建模块就不改现有模块
- **渐进优于颠覆** — 一次改一个点，验证有效了再改下一个
- **记录优于口头** — 改了什么、为什么改、谁决定的，写在变更日志里

## 四、设计理念溯源

以下是构建本系统过程中提炼出的核心认知。

**这个列表不是封闭的。** 每次项目实践、每次外部学习、每次失败教训，都可能产生新的设计认知。发现了新认知 → 追加到这个列表，标注来源和日期。

1. **约束三级火箭**— 建议→文档→代码，逐级升级约束力度
2. **验证 Agent 黑盒验证**— 独立验证，不信任执行者的自我报告
3. **渐进式信息披露**— AGENTS.md 当目录而非百科全书
4. **仓库知识是唯一真相源**— 不在仓库里 = 不存在
5. **大地图 + 小地图**— brief.md 是全局视图，task-spec 是局部视图 + 指针
6. **打卡机制**— task-spec 状态追踪是唯一真相，不靠对话记忆
7. **冷启动恢复**— 每次从文件恢复，假设上一次的记忆已完全丢失
8. **上下文裁剪**— 只给结论不给过程，宁少勿多
9. **反馈学习闭环**— 项目完成后互相反馈，协作质量随时间提升
10. **名片与档案**— 每个执行者一份档案，记忆丢了也能快速恢复合作模式
11. **消息即文档**— 消息脱离上下文也能被理解
12. **平台无关**— 底层方法论不绑定任何平台，工具会变方法不变
13. **事前预判**— 每个环节开始前预判风险，不只是事后恢复
14. **元能力培养**— 不只教"怎么做"，还教"怎么面对不会做的事"



15. **拆分粒度的黄金法则**— 拆分的目的不是文件越多越好，而是"恰当时机能一次性调取所需全部信息"。文件分开存储是为了结构化管理，但链路引用让智能体可以在需要时一次性加载多个模块。**判断标准：如果两块内容总是一起被需要，合并比拆分好；如果有些时候只需要其中一块，拆分比合并好。** 宁可在一次调取时多读 1KB，也不要因为拆分导致关键信息缺失而做出错误判断。
16. **全局控制者的双层视野**— 派活方不只是"管项目的人"，而是同时监督两个层面的全局控制者：**系统层**（系统规则是否需要改进、约束是否需要升级、模板是否够用）和**项目层**（每个任务什么状态、谁在做什么、下一步推进什么）。两层视野缺一不可：只看项目会忽略系统腐化，只看系统会忽略项目停滞。
17. **系统规则与系统数据分离**— `system/` 下有两类文件：**规则文件**（docs/、templates/、README.md）定义"怎么做"，修改需审批；**数据文件**（resource-profiles.md 等）记录运行时信息，正常读写。数据文件是"外部插口"——可插拔、可扩展（按需创建新主题）、可重建（全部删了系统照常运作，下次运行自动新建空文件）。这跟大型软件系统的设计一样：配置文件（规则）和数据库（数据）分离，数据库清空不影响程序运行，只是丢了历史数据。
18. **不怕碎片化，关键是结构化和整合能力**— 碎片化不是问题，结构化才是答案。文件可以很多很碎，只要有清晰的索引（速查表）、有效的链路引用（场景调取清单）、和明确的使用时机（每个文件在什么场景被谁读），碎片就变成了模块。智能体的价值不在于记住所有东西，而在于知道在什么时候去找什么东西。
19. **垃圾回收对抗熵增**— 系统运行久了，状态会腐化。周期性健康扫描不是可选的，而是系统长期存活的必要条件。不是等出了问题才修，而是定期主动发现问题。
20. **质量是流水线，不是检查点**— 每个阶段都有明确的产物标准，未达标不能进入下一阶段（强制结构化中间产出）。验证者与执行者分离（独立验证角色）。需求定义是最重要的阶段，70% 的质量问题源于需求不清。
21. **状态转换有守卫条件**— 任务状态不是随便改的标签，每次转换都有必须满足的前置条件。违反守卫条件 = 转换无效。这把"流程规范"从建议变成了硬约束。
22. **修系统优于修任务**— 遇到问题时，直接修当前任务 = 一次性收益；改系统（规则、模板、检查清单）= 后续所有任务受益。这是复利来源。当一个问题反复出现时，优先问"系统缺了什么"，而不是一次次手工补锅。
23. **交付前必须自检**— 执行者完成任务后不能凭感觉"做完了"，必须逐条对照验收标准自检。返工时不能只看反馈修表面，必须重新阅读原始需求回到源头。
24. **自维持靠文件冗余，不靠单点**— 系统的持续运转不依赖任何单一文件或机制。Heartbeat 没更新？冷启动恢复兜底。签到表漏了？task-spec 状态兜底。沟通记录没写？task-spec 指派字段兜底。多层冗余确保任何一层断裂都不会让系统崩溃。已知限制：消息发送和文件更新不是原子操作，极端情况下可能重复分发，靠执行者端的去重意识和项目签到表防线缓解。
25. **防断联是预防，冷启动是治疗，两者都不能少**— agent 不能等断了再恢复，要在工作中主动设保险（更新 Heartbeat、设定时器、写中间记录）。三种工具各有适用场景：Heartbeat 做常规巡检，定时器做精确唤醒，中间记录做进度保存。这套机制适用于所有 agent 角色，不只是派活方。
26. **系统是引导，不是枷锁 — 你有大脑，用它**— 系统不可能覆盖所有场景。agent 必须具备在系统纪律框架内自主解决未覆盖问题的元认知能力：主动利用平台已有工具为自己服务（定时、唤醒、文件记录），遇到新情况先思考再行动再记录。系统教的是思维方式和工作纪律，不是一本操作手册让你逐条对照执行。红线是系统规则不能自改，红线内自由发挥。
27. **反思是系统进化的引擎，不是可选项**— 项目完成后强制反思五问（流程/工具/协作/预判/系统），从中提取系统改进建议。"记下来"只是存储，"问自己该改什么"才是进化。个人经验留在经验库，系统级改进提交审批后改系统。每个项目都是系统进化的养料。
