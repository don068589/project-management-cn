# 自循环保障机制

> 解决元问题：谁来确保系统本身在运行？谁来确保 agent 不会忘记遵守规则？
> 本文件是系统的"免疫系统"，确保自循环不断裂。

---

## 一、元问题与解决方案

### 问题 1：agent 怎么知道要读操作手册？

**解决方案（按优先级）：**

1. **skill 注入（最佳）** — 如果平台支持 skill 机制，配置触发条件：
   - 收到任务规格书路径 → 自动提醒读 `executor.md`
   - 被要求立项/拆解/评审 → 自动提醒读 `coordinator.md`

2. **派活消息中写明（兜底）** — 如果没有 skill 机制，在派活消息里直接写：
   ```
   请执行任务。规格书在 [路径]。
   执行前先读 system/docs/executor.md（执行方操作手册）。
   ```

3. **README.md 入口引导（最终兜底）** — 任何人拿到系统，读 README.md 就知道该看哪个文件（快速入门节）。

**不依赖 agent "记得"要读手册。** 三重机制确保：skill 自动注入 → 消息显式写明 → README 入口引导。

### 问题 2：agent 怎么知道要更新签到表？

**解决：操作手册中的检查清单。**

coordinator.md 的分发流程写着：
```
分发前检查：
- [ ] 项目签到表无冲突
分发后立即更新：
- task-spec 状态 + 沟通记录 + 项目签到表登记
```

这不是"建议"，是流程的一部分。agent 按手册走，自然会执行。

### 问题 3：Heartbeat 怎么知道要更新？

**解决：runtime.md 的"Heartbeat 动态更新"机制。**

每个阶段完成后，更新 Heartbeat 内容。这写在 runtime.md 里，是标准流程的一部分。

### 问题 4：中间出了岔子怎么办？

**解决：中间事件记录表（runtime.md）+ 决策指南（decision-guide.md）。**

- 出了什么事 → 查 runtime.md 的"中间事件记录"表，知道记在哪里
- 不知道怎么决定 → 查 decision-guide.md，有判断标准

### 问题 5：谁来确保 agent 遵守了这些规则？

**解决：三重保障。**

1. **skill 注入**（被动） — agent 打开项目相关功能时，skill 自动提醒关键规则
2. **检查清单**（主动） — 操作手册中的 checklist，agent 自查
3. **检查脚本**（自动） — `tools/` 下的脚本，机械检查状态一致性

这就是约束三级火箭（governance.md）的应用：
- L1：skill 注入提醒
- L2：操作手册的流程规则
- L3：检查脚本自动拦截

## 二、双层自循环

**你是全局控制者。** 你同时监督两个层面：
- **项目层** — 每个任务什么状态、谁在做什么、下一步推进什么
- **系统层** — 系统规则是否需要改进、约束是否需要升级、模板是否够用

不是"有空才看系统"，而是两层始终在视野内。项目层是高频日常，系统层是低频但同样重要。

### 项目层循环（每次心跳 / 每次对话）

```
agent 醒来
    │
    ▼
读 Heartbeat → 知道要检查哪些项目
    │
    ▼
冷启动恢复（runtime.md）→ 扫描文件恢复项目状态
    │
    ▼
发现项目待办 → 读对应操作手册 → 执行
    │
    ├─ 分发任务 → 更新 task-spec + 项目签到表 + 沟通记录
    ├─ 评审任务 → 写 review-record + 更新状态
    ├─ 处理中间事件 → 记录到对应位置（runtime.md 指示）
    │
    ▼
环节完成 → 更新 Heartbeat 检查项（写下一步做什么）
    │
    ▼
等待下一次心跳 / 对话 → （循环）
```

### 系统层循环（每完成一个项目 / 每月一次）

```
项目完成
    │
    ▼
经验沉淀（knowledge.md）→ 提炼教训
    │
    ├─ 可复用规则 → 升级到系统文件（governance.md 扩展流程）
    ├─ 反复问题 → 约束升级（L1→L2→L3）
    ├─ 新任务类型 → 更新 quality.md 适配表
    ├─ 执行者表现 → 更新 resource-profiles.md 名片
    │
    ▼
系统复盘检查（governance.md）→ 流程顺畅？模板够用？返工率高？
    │
    ▼
更新系统文件 → 记录变更日志
    │
    ▼
下一个项目受益 → （循环）
```

### 两层循环的关系

- **项目层**是高频循环（每次心跳），关注"项目进展到哪了，下一步做什么"
- **系统层**是低频循环（每完成一个项目），关注"系统本身是否需要改进"
- 项目层产生的问题和教训，喂入系统层；系统层的改进，惠及下一个项目层

**Heartbeat 中的系统级检查项：**
```markdown
## 系统维护（低频，每月检查一次）
- 上次系统复盘是什么时候？超过 1 个月了吗？
- improvements.md 待改进清单有积压吗？
- 有没有约束需要从 L1 升级到 L2/L3？
```

**关键：两层循环都不依赖记忆，都靠文件驱动。**

## 三、断裂检测

### 什么情况下自循环会断裂？

| 断裂点 | 症状 | 修复方式 |
|--------|------|----------|
| agent 没读操作手册 | 跳过流程步骤 | skill 提醒触发 |
| 签到表没更新 | 重复分发 | dashboard.py 检查状态不一致 |
| Heartbeat 没更新 | 下次心跳不知道做什么 | 冷启动恢复流程兜底 |
| 索引没更新 | 找不到项目 | Heartbeat 冷启动时自检 |
| 沟通记录没写 | 恢复时不知道跟谁聊过 | 评审时检查沟通记录完整性 |
| 中间事件没记录 | 决策依据丢失 | 评审时发现缺失，补记 |

### 兜底机制

即使某个记录遗漏了，系统也不会崩溃，因为有**多重恢复路径**：

1. **task-spec 状态字段** — 最基础的恢复信息，只要状态对，就能接上
2. **项目签到表** — 交叉验证，知道谁在做什么
3. **沟通记录** — 恢复通信上下文
4. **Heartbeat** — 知道下一步做什么
5. **冷启动恢复流程** — 终极兜底，从零开始扫描文件恢复

丢失其中任何一个，都能通过其他路径恢复。全部丢失的概率几乎为零（因为分布在不同文件中）。

## 四、时机→文件 速查表

**agent 在任何时刻，只需要查这张表就知道该读什么：**

| 我现在要做什么 | 读什么 |
|---------------|--------|
| 第一次接触这个系统 / 想了解设计理念 | `philosophy.md` |
| 刚醒来，不知道有什么要做 | `runtime.md` 冷启动恢复流程 |
| 要立项 / 拆解 / 分发 / 评审 | `coordinator.md` |
| 收到任务规格书，要执行 | `executor.md` |
| 不知道怎么判断员工交付质量 | `decision-guide.md` 评审决策指南 |
| 不知道怎么写 task-spec / 评审记录 | `decision-guide.md` 文件编写指南 |
| 不知道怎么跟某类执行者沟通 | `communication.md` 沟通方式矩阵 |
| 面对不认识的执行者/平台 | `philosophy.md` 元能力：探索未知系统 → 探索后创建名片到 `resource-profiles.md` |
| 要分发任务给某个执行者 | `resource-profiles.md` 读对方名片（能力/偏好/调用方式） |
| 项目完成后给执行者反馈 | `communication.md` Feedback 类型 → 然后更新 `resource-profiles.md` 名片 |
| 不知道中间事件记在哪 | `runtime.md` 中间事件记录表 |
| 任务出问题了不知道怎么处理 | `scheduling.md` 异常处理与升级 |
| 项目要暂停 / 取消 / 恢复 | `lifecycle.md` |
| 系统本身要修改 / 加新模块 | `governance.md` 系统扩展流程 |
| 想了解系统为什么这样设计 | `philosophy.md` 设计理念溯源 |
| 不确定系统哪里可能断裂 | 本文件（self-loop.md）|
| 系统该复盘了 / 约束要升级 | `governance.md` 系统复盘 + 约束三级火箭 + 垃圾回收 |
| 需要独立验证者评审 | `quality.md` 独立验证角色 |
| 任务状态转换不确定 | `task-management.md` 任务状态机 + 守卫条件 |
| 长任务执行中怕断联 | `runtime.md` 防断联三步法 |
| 需要等一会儿再检查 | `runtime.md` 定时器 vs Heartbeat 选择指南 |

**原则：这张表是 agent 的"总导航"。不确定该看什么时，先看这张表。**

## 五、场景调取清单（一次性加载）

**拆分是为了管理，引用是为了使用。** 文件分开存储但通过引用链路一次性调取。

以下是每个工作场景需要**一次性读取**的文件清单。agent 在进入某个场景时，按清单一次性加载所有需要的文件，而不是"边做边查"：

| 工作场景 | 一次性加载 | 总大小 | 说明 |
|----------|-----------|--------|------|
| **拆解任务** | coordinator.md + task-management.md + quality.md（验收标准部分） | ~8KB | 依赖、量化标准都在手边 |
| **分发任务** | coordinator.md（分发节） + resource-profiles.md（目标执行者名片） + communication.md（Ask 格式） | ~6KB | 知道派给谁、怎么说 |
| **评审任务** | coordinator.md（评审节）+ quality.md + decision-guide.md（评审决策） + task-spec + 产出文件 | ~8KB+产出 | 判断标准和决策指南都在手边 |
| **处理变更** | coordinator.md（变更节）+ decision-guide.md（brief 调整）+ change-request 模板 | ~5KB | 评估影响+写变更请求 |
| **紧急任务** | scheduling.md（紧急通道节）+ task-spec 模板 | ~4KB | 快速通道流程 |
| **项目完成** | coordinator.md（验收节）+ knowledge.md + communication.md（Feedback）+ resource-profiles.md | ~10KB | 经验沉淀+反馈+名片更新 |
| **冷启动恢复** | runtime.md + data_structure.md + status.md（如有） | ~3KB | 最小恢复集 |
| **执行任务** | executor.md + task-spec | ~5KB | 执行者只需要这两个 |

**关键设计理念：文件分开存是为了"不用的时候不占空间"，引用清单是为了"要用的时候一次到位"。**

**这不矛盾。** 拆分解决的是存储和维护问题（改一个不影响其他），引用解决的是使用问题（需要时一次性获取）。两者通过链路引用连接。

## 六、系统引导文件的层次结构

```
README.md                    ← 系统大纲（入口）
    │
    ├─ docs/philosophy.md    ← 设计理念与元能力（系统灵魂）
    │
    ├─ docs/coordinator.md   ← 派活方"怎么做"
    ├─ docs/executor.md      ← 执行方"怎么做"
    │
    ├─ docs/runtime.md       ← "醒来后做什么"
    ├─ docs/decision-guide.md ← "怎么做决定"
    ├─ docs/communication.md  ← "怎么跟人说话"
    ├─ docs/self-loop.md      ← "怎么确保不断裂"
    │
    ├─ docs/task-management.md ← 规则细节（按需查）
    ├─ docs/quality.md
    ├─ docs/scheduling.md
    ├─ docs/lifecycle.md
    ├─ docs/collaboration.md
    ├─ docs/knowledge.md
    └─ docs/governance.md
```

**理念层（系统灵魂）：** philosophy.md — 为什么这样设计、怎么面对未知
**入口层（必读）：** README.md → 指向操作手册
**操作层（按角色读）：** coordinator.md 或 executor.md
**保障层（按需查）：** runtime / decision-guide / communication / self-loop
**规则层（遇到具体问题查）：** 7 个主题文件

**agent 不需要一次读完所有层。skill 会在合适的时机提醒该读哪个。**

## 七、是否需要自动化脚本？

**当前判断：文件机制足够，暂不需要额外脚本。**

理由：
1. 文件读写是 agent 的原生能力，不需要额外工具
2. 检查清单嵌入在操作手册中，跟着流程走自然覆盖
3. dashboard.py 已经提供了状态一致性检查

**什么时候需要加脚本（L3 约束升级）：**
- 同一个遗漏反复出现 3 次 → 写检查脚本自动拦截
- 例如：agent 反复忘记更新签到表 → 写脚本在分发时自动检查项目签到表

这遵循约束三级火箭原则：先靠文档，不行再上代码。
