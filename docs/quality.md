# 质量保障机制

> 质量门禁、评审标准、验证流程、迭代闭环。
> 核心理念：**质量是一个流水线，不是一个检查点。**

---

## 强制结构化中间产出

_来源：MetaGPT — "每个阶段必须有符合标准的产物才能继续"_

**问题：** 如果中间产出没有结构化标准，错误会随流水线放大（幻觉传播）。

**规则：每个阶段都有明确的产物标准，未达标不能进入下一阶段。**

| 阶段 | 必须产出 | 产出标准 | 检查人 |
|------|---------|---------|--------|
| 立项 | brief.md | 目标、范围、角色、里程碑全填 | 决策者 |
| 拆解 | task-spec（全部） | 验收标准量化、依赖标注、输入输出明确 | 派活方自查 |
| 执行 | 产出文件 | 在 task-spec 指定位置、格式符合要求 | 执行者自查 |
| 评审 | review-record | 逐条对照验收标准、有结论 | 派活方 / 验证者 |
| 验收 | final-report | 所有任务已验收、经验教训已填 | 决策者 |

**阶段门禁：** 上一阶段的产出不符合标准 → 不能开始下一阶段。这是硬约束，不是建议。

## 独立验证角色

_来源：质量门禁流水线 — "验证 agent 与开发 agent 分离"_

**问题：** 执行者自己检查自己的产出，存在盲点。

**规则：重要任务的验证必须由非执行者完成。**

| 任务重要性 | 验证方式 |
|-----------|---------|
| 低（简单修改、格式调整） | 派活方直接评审 |
| 中（标准开发、调研） | 派活方评审 + 抽查 |
| 高（核心功能、架构变更、数据处理） | **独立验证者**评审 |

**独立验证者的要求：**
- 不是执行该任务的人
- 只拿到产出文件和验收标准（黑盒验证）
- 不需要了解执行过程
- 验证结果记录在 review-record

**怎么找独立验证者：**
- 另一个 agent / subagent
- 另一个执行者
- 决策者本人
- 关键是"不是执行者自己"

**独立验证的操作流程：**

1. 派活方决定需要独立验证 → 写一个验证任务的 task-spec（可简化）
2. 验证 task-spec 的"输入"只给两样东西：**产出文件路径** + **原任务的验收标准**
3. 验证 task-spec 的"验收标准"就是：逐条对照原任务验收标准检查，输出 review-record
4. 派活消息用 Ask 格式，附加说明："这是验证任务，你不需要了解执行过程，只需对照标准检查产出"
5. 验证者提交 review-record → 派活方根据结论决定通过/返工

**简化验证（小任务）：** 如果任务本身很小、验收标准 ≤3 条，可以不写验证 task-spec，直接在 Ask 消息中写明产出路径和验收标准让验证者检查。

**防偷懒铁律（写进验证任务规格书）：**
1. 具体标准 — 不说"确保能运行"，说"运行完整测试并报告所有失败项"
2. 全面检查 — 强制覆盖边界情况
3. 反向测试 — 输入应该失败的数据，确认系统真的报错
4. 显式指令 — "必须逐条检查所有验收标准才能标记为通过"

## 质量门禁

任务从"待评审"到"已验收"，**必须有评审记录**（review-record）。

| 阶段 | 门禁 | 通过条件 |
|------|------|----------|
| 立项 → 拆解 | 需求确认门禁 | 决策者确认需求，明确说"可以开始" |
| 执行中 | 中间产出门禁 | 用户确认点全部通过（如有）|
| 待评审 → 已验收 | 评审门禁 | 评审记录存在且结论为"通过" |
| 项目验收 | 最终门禁 | 所有任务已验收 + 决策者拍板 |

跳过评审需决策者明确豁免（在 task-spec 状态追踪中注明）。

## 验收标准可衡量性

写验收标准时，尽量量化：
- ❌ "输出格式清晰" → ✅ "JSON 格式，含 name/value/timestamp 字段"
- ❌ "性能好" → ✅ "响应时间 < 2 秒"
- ❌ "代码质量高" → ✅ "无语法错误，有异常处理，关键函数有注释"

无法量化的标准（审美、风格等）→ 标为"用户确认点"，由决策者判断。

## 任务类型适配

不同任务类型的评审重点不同：

| 任务类型 | 示例 | 评审重点 |
|----------|------|----------|
| 代码开发 | 建项目、写功能、修 bug | 功能完整性、代码质量、测试覆盖 |
| 调研搜索 | 市场调研、竞品分析 | 信息准确性、覆盖面、结论可靠性 |
| 数据采集 | 爬取、监控、信息收集 | 数据完整性、格式规范、去重 |
| 工作流制作 | 自动化、脚本、定时任务 | 可靠性、异常处理、可维护性 |
| 内容创作 | 文档、PPT、视频脚本 | 内容质量、排版、受众适配 |
| 设计/UI | 界面、交互、视觉 | 审美、一致性、可用性 |
| 系统运维 | 配置、部署、监控 | 安全性、稳定性、可回滚 |

_评审时根据任务类型选择合适的评分维度。_

## 自动化验证

可自动验证的产出，优先用工具验证：

| 产出类型 | 自动验证方式 |
|----------|-------------|
| 代码/脚本 | 实际运行，检查输出 |
| Web/UI | 用浏览器工具截图、点击测试 |
| 数据文件 | 检查行数、格式、抽样验证 |
| API/接口 | 发请求验证返回值 |
| 文档 | 检查结构完整性、链接有效性 |

_自动验证不替代人工评审，但能提前发现明显问题。_

## 交付验证

评审前第一步：验证产出文件是否存在于规格书指定位置。
- 文件不存在 → 不开始评审，直接反馈
- 文件存在但格式不对 → 记入评审记录

## 评估优化循环

返工是标准闭环迭代：生成 → 评估 → 优化 → 重新评估。

- 每次返工是一个迭代轮次，评审记录标注轮次编号
- 每轮记录：第几轮、上轮反馈、这轮改了什么、还差什么
- **超过 3 轮升级** — 提供 3 轮改进记录、卡住原因、替代方案建议

## 返工失败后换方案

超过 3 轮仍不达标，升级后决定换方案：

1. 当前任务标记为"已放弃"，记录原因
2. 新建替代任务 TASK-XXX-v2，引用原任务
3. 说明：原方案为什么失败、新方案是什么
4. 可以换执行者、换工具、换实现方式
5. 原任务记录保留，作为经验参考

## 变更后已验收任务的处理

- **小改（≤30% 工作量）：** 重新打开旧任务，修改后重新评审
- **大改（>30%）：** 新建任务，旧任务保持已验收
