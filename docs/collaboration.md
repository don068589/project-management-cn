# 协作与沟通机制

> 汇报、确认、反馈、通知、状态纪律、派活工具。

---

## 汇报机制

| 场景 | 动作 |
|------|------|
| 日常调度（分发、催进度） | 派活方自主决定 |
| 评审结论（通过/返工） | 派活方自主决定 |
| 项目立项 | 汇报决策者审批 |
| 方向性选择 / 重大变更 | 汇报决策者拍板 |
| 里程碑完成 | 汇报决策者知悉 |
| 最终验收 | 汇报决策者拍板 |

## 需求确认机制

- 立项后、开工前，必须与决策者反复确认需求
- brief.md 有"需求确认记录"表，记录每轮确认
- 决策者明确说"可以开始"才进入拆解
- 需求不清晰的地方主动提问，不要猜

**确认内容：** 目标理解、范围边界、验收标准、隐含偏好/约束。

## 中间产出确认机制

涉及主观判断（审美、风格、精度等）的产出，派活方无法替代决策者判断。

**规则：**
- task-spec 有"用户确认点"字段
- 涉及主观判断 → 用 confirmation-record 提交决策者
- 确认不通过 → 记录反馈，调整后重新提交

**关键原则：宁可多确认一次，不要做完了才发现方向错了。**

## 执行者反向反馈机制

执行者发现以下情况，应主动反馈派活方：
1. 前置任务产出有问题
2. 规格书要求不可行或有歧义
3. 发现规格书没提到的风险
4. 预计无法按时完成
5. 需要额外输入或权限
6. **发现新需求/机会** — 执行过程中发现"这个项目还应该做 X"，但 X 不在当前任务里

**派活方收到反馈后：**
- 前置有问题 → 重新打开修复
- 有歧义 → 澄清并更新规格书
- 新风险 → 更新风险登记册
- 无法按时 → 调整截止时间或换方案
- 需额外输入 → 提供或升级

## 规格书更新通知

- 修改规格书后必须通知执行者
- 通知内容：改了什么、为什么改、对执行的影响
- 重大修改 → 执行者确认收到后才继续
- 在沟通记录中记录

## 决策者最终否决权

- 决策者可否决任何已验收的任务和已完成的项目
- 否决后按变更请求或返工流程处理
- 为减少否决：关键产出先走中间产出确认

## 状态更新纪律

每次状态变更必须立即更新 task-spec 中的状态字段和状态追踪表。

| 状态变更 | 触发点 | 谁更新 |
|----------|--------|--------|
| 待分发 → 进行中 | 分发时 | 派活方 |
| 进行中 → 阻塞中 | 执行者报告阻塞 | 派活方 |
| 阻塞中 → 进行中 | 阻塞解除 | 派活方 |
| 进行中 → 待评审 | 执行者交付时 | 派活方 |
| 待评审 → 返工 | 评审不通过 | 派活方 |
| 返工 → 待评审 | 执行者重新提交 | 派活方 |
| 待评审 → 已验收 | 评审通过 | 派活方 |
| 进行中 → 已暂停 | 项目暂停 | 派活方 |
| 已暂停 → 进行中 | 项目恢复 | 派活方 |
| 任何 → 已取消 | 项目取消 | 派活方 |
| 任何 → 已放弃 | 换方案时 | 派活方 |

## 派活方不亲自执行

- 派活方负责规划、拆解、分发、评审、验收
- 具体执行必须分发给执行者
- 唯一例外：极其简单的任务（如改一行配置），在状态追踪中标注"自行执行"

## 派活工具使用规范

派活方通过当前可用的工具分发任务并追踪沟通。

**通用流程（与平台无关）：**
1. 选择工具 → 查 `resource-profiles.md` 中当前可用的通信工具
2. 发送派活消息 → 按 `communication.md` 的消息格式
3. 记录通信标识 → 写入 task-spec 沟通记录（Thread ID、session 标识等）
4. 后续追踪 → 通过同一标识继续沟通

**具体工具的调用方式记录在 `resource-profiles.md` 的"当前通信基础设施"中。**
