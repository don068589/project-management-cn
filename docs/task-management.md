# 任务管理机制

> 任务状态机、认领防重复、依赖管理、写前冲突检测。

---

## 任务状态机

_来源：A2A Protocol 任务生命周期 + 实践沉淀_

每个任务有且仅有以下状态，状态转换有严格的守卫条件：

```
待分发 → 进行中 → 待评审 → 已验收
                      ↓
                    返工 → 待评审（循环）
                      ↓（超过 3 轮）
                   已放弃 → 替代任务（新建）

特殊状态：
待分发 → 已取消（项目取消时）
进行中 → 已暂停（项目暂停时）
已暂停 → 进行中（项目恢复时，可能换执行者）
进行中 → 阻塞中（等待输入/依赖/决策者确认）
阻塞中 → 进行中（阻塞解除后）
```

### 状态转换守卫条件

| 转换 | 守卫条件（必须满足才能转换） | 谁触发 |
|------|---------------------------|--------|
| 待分发 → 进行中 | 前置依赖已就绪 + 项目签到表无冲突 + 执行者已分配 | 派活方 |
| 进行中 → 阻塞中 | 执行者报告阻塞原因 | 执行者反馈 |
| 阻塞中 → 进行中 | 阻塞原因已解决 | 派活方确认 |
| 进行中 → 待评审 | 产出文件存在于指定位置 | 执行者交付 |
| 待评审 → 已验收 | review-record 存在且结论为"通过" | 派活方 |
| 待评审 → 返工 | review-record 存在且结论为"不通过" | 派活方 |
| 返工 → 待评审 | 修改完成 + 产出文件已更新 | 执行者重新交付 |
| 返工（第 3 轮）→ 已放弃 | 提供 3 轮记录 + 卡住原因 + 替代方案 | 派活方升级 |

**违反守卫条件 = 状态转换无效。** 例如：产出文件不存在就不能标为"待评审"。

## 任务依赖管理

- 每个 task-spec 有"依赖关系"字段，标明前置和后续依赖
- 分发时按依赖顺序调度：前置未完成 → 后续不分发
- 无依赖的任务可以并行分发
- 前置完成后，立即检查并分发后续任务

**依赖状态：**
- 已就绪 — 前置已验收，产出可用
- 未就绪 — 前置未完成，等待中

## 循环依赖检测

- 拆解完所有任务后，画出依赖图检查有无环
- 从每个任务出发沿依赖链走，走回自己就是环
- 发现环 → 重新拆解，打破循环
- 分发前再次确认：所有"待分发"任务的前置要么已就绪要么无依赖

## 任务嵌套（子任务拆解）

- 大型任务可拆解为子任务
- 命名：`TASK-003a`、`TASK-003b`（父编号后加字母）
- 子任务各自有独立的 task-spec
- 父任务状态取决于子任务：全部验收 → 父验收
- 嵌套最多两层，超过说明拆解粒度有问题

## 防止重复分发

- **分发前必须检查任务状态** — 已"进行中"或之后的状态，不要再分发
- 分发时立即更新状态为"进行中"，沟通记录写入通信标识
- 重启后恢复上下文：
  1. 跑 dashboard.py 查看所有任务状态
  2. "进行中"的 → 检查沟通记录中的通信标识，尝试联系确认进度
  3. 无法联系 → 按异常处理流程处理（见 `scheduling.md`）

## 任务认领机制

**背景：** 来自真实事故 — 两个 agent 的 subagent 同时转录了同一个视频，产生重复文件。

**机制：** 每个项目目录下有自己的 `task-registry.md`（项目级签到表），记录本项目正在执行的任务。签到表是项目内部的防重复机制，不是跨项目的全局锁。跨项目防重复靠派活方的调度职责保障。

**规则：**
- 每个任务分配唯一 ID（TASK-001、TASK-002……）
- 开始前查项目签到表 — 同样任务已认领且未过期 → 不重复执行
- 认领时写入：任务ID、认领人、状态、开始时间、过期时间、产出位置
- 状态流转：`认领中 → 进行中 → 已完成 / 已释放`
- 过期机制：默认 2 小时，可自定义。过期后他人可接手（标注"接替"）
- 已完成记录保留 24 小时后可清理

**派活方职责：** 分发时登记，完成时更新，定期清理。

**签到表格式（项目目录下的 task-registry.md）：**

```markdown
# 任务签到表

> 项目数据文件 — 正常读写，无需审批

| 任务ID | 认领人 | 状态 | 开始时间 | 过期时间 | 产出位置 | 备注 |
|--------|--------|------|----------|----------|----------|------|
| TASK-001 | Forge | 进行中 | 03-21 10:00 | 03-21 12:00 | tasks/ | |
```

状态值：`认领中` / `进行中` / `已暂停` / `已完成` / `已释放`

## 写前冲突检测

往项目目录写文件前，必须三步检查：

1. **文件存在性** — 已存在 → 对比内容，相同跳过，不同改名保存
2. **签到表检查** — 查项目签到表，有人正在做同样任务 → 不要开始，反馈派活方
3. **内容相似性** — 目标目录下有高度相似文件 → 暂停，反馈确认

**原则：宁可多问一次，不要产生重复文件。**

## 并行任务上限

- 同一执行者最多 **2 个** 进行中任务（硬限制）
- 同一项目最多 **5 个** 并行进行中任务
- 超出 → 排队等待，状态保持"待分发"
- 紧急任务不受此限制，但会暂停低优先级任务
- 排队策略：高优先级优先出队，同优先级先进先出
