# 知识与信息管理

> 知识沉淀、知识即仓库、渐进式信息披露。

---

## 知识沉淀

项目完成后，经验教训必须提炼和归档。

**存储位置（全部在 /path/to/projects/ 内）：**

| 经验类型 | 存储位置 | 说明 |
|----------|---------|------|
| 单个项目的经验 | `/path/to/projects-data/[项目名]\final-report.md` | 项目内部，跟项目走 |
| 跨项目通用经验 | `/path/to/projects-data/lessons\[主题].md` | 多个项目都适用的教训 |
| 执行者合作经验 | `system/resource-profiles.md`（名片） | 系统数据文件，正常更新 |
| 系统规则改进建议 | `system/improvements.md` 待改进清单 | 建议正常写入，改规则文件需决策者审批 |
| 可复用脚本/工具 | `/path/to/projects-data/shared-tools\` | 审批后才进 `system/tools/` |

**核心规则：**
- 经验记录是正常操作 — 名片、lessons、final-report 随时可写
- 经验**变成系统规则**才需要审批 — 改 docs/、templates/ 需要决策者拍板
- 所有内容都在 `/path/to/projects/` 内部，不涉及外部系统

**沉淀流程：**
1. 项目验收时，回顾整个过程
2. 提炼经验教训，写入 final-report
3. 有跨项目通用价值 → 写入 `/path/to/projects-data/lessons\[主题].md`
4. 合作经验 → 更新 resource-profiles.md 名片（积累规则、合作记录）
5. 发现系统规则可改进 → 写入 `improvements.md` 待改进清单（含具体建议和理由）
6. 决策者审批后 → 更新系统规则文件

## 反思提炼流程

> 经验沉淀解决"记下来"，反思提炼解决"从经验里挖出系统改进点"。
> 这是系统自我进化的核心引擎。

### 什么时候做反思

- 每个项目验收后（必做，嵌入 final-report 流程）
- 系统复盘时（governance.md 定义的周期）
- 遇到重大事故或反复出现的问题时（随时）

### 反思五问（结构化自问清单）

做完一个项目（或处理完一个事故），强制问自己这五个问题：

**1. 流程问题：** 有没有哪个环节卡住了、走弯路了、或者跳过了？
   - 卡住 → 系统对这个场景的引导够不够？缺指南还是指南不清楚？
   - 走弯路 → 是信息不足还是判断失误？信息不足说明信息披露机制有漏洞
   - 跳过 → 是合理简化还是偷懒？如果是偷懒，约束够不够强？

**2. 工具问题：** 有没有想用某个工具但不知道怎么用、或者发现了更好的用法？
   - 不知道怎么用 → 名片/环境经验记录该更新了
   - 发现更好用法 → 记录到 resource-profiles.md 第八节（经验记录）

**3. 协作问题：** 执行者理解任务规格书有困难吗？返工次数超预期吗？
   - 理解困难 → task-spec 模板缺字段？沟通协议缺消息类型？
   - 返工超 2 轮 → 验收标准不够清晰？还是执行者能力不匹配？

**4. 预判问题：** 有没有什么事后看是可以提前预见的？
   - 风险没识别到 → 风险登记册机制够不够？
   - 依赖关系遗漏 → 拆解阶段的检查清单够不够？

**5. 系统问题：** 如果再做一次同类项目，系统需要改什么才能做得更好？
   - 这是最关键的问题 — 不是"我下次注意"，而是"系统改什么让任何人下次都不犯"

### 从回答到改进建议

反思完五个问题后，把答案分类：

| 答案类型 | 处理方式 | 审批 |
|----------|---------|------|
| 个人经验（"我学到了..."） | 写入 final-report 或 lessons/ | 不需要 |
| 名片更新（"这个执行者擅长/不擅长..."） | 更新 resource-profiles.md | 不需要 |
| 环境经验（"这个工具有个坑..."） | 更新 resource-profiles.md 第八节 | 不需要 |
| **系统改进建议**（"模板应该加个字段"、"检查清单漏了一项"） | 写入改进建议，提交决策者 | **需要** |

### 改进建议的写法

```markdown
### 改进建议 #[编号]

- **来源项目：** [项目名]
- **发现的问题：** [具体描述]
- **影响范围：** [只影响本类项目 / 影响所有项目]
- **建议改动：** [改哪个文件的哪个部分，怎么改]
- **理由：** [为什么这样改能防止问题再次发生]
- **风险：** [改了会不会影响其他机制]
```

### 改进建议的去向

1. 写入 `improvements.md`（待改进清单）
2. 决策者审批后 → 执行改动 → 更新变更日志
3. 决策者不批 → 记录理由，留在清单里供将来参考

### 与现有机制的关系

```
项目完成 → final-report（记经验）
              ↓
         反思五问（挖改进点）
              ↓
         ┌─ 个人经验 → lessons/
         ├─ 合作经验 → 名片
         ├─ 环境经验 → 第八节
         └─ 系统改进 → improvements.md → 决策者审批 → 改系统
              ↓
         系统复盘时批量审阅 improvements.md
```

## 知识即仓库

**原则：如果不在项目目录里，对下一个接手的人来说就不存在。**

- 所有影响项目走向的决策，必须落盘到项目目录内
- 聊天/消息中达成的共识 → 记录到 task-spec 沟通记录或 brief.md
- 口头协商的方案 → 写进相关文档

**检查时机：**
- 每次里程碑评审时，检查关键决策是否已落盘
- 项目交接时，确认新负责人只需读项目目录就能完全接手

## 渐进式信息披露

**问题：** 把所有项目信息一次性塞给执行者 → 上下文膨胀 → 抓不住重点。

**规则：**
- **brief.md 是大地图** — 项目全局（目标、范围、角色、风险），执行者可选读
- **task-spec 是小地图** — 只包含该任务需要的信息 + 指针
- 执行者只需读自己的 task-spec 就能开始工作
- 不需要的信息不塞进 task-spec

**task-spec 的"信息指针"字段：**

```markdown
## 相关上下文（按需查阅）

| 主题 | 位置 | 何时需要 |
|------|------|----------|
| 项目整体目标 | brief.md | 需要理解大方向时 |
| 前置任务产出 | tasks/TASK-001-xxx.md 的输出 | 需要使用前置产出时 |
| 技术方案讨论 | brief.md 需求确认记录 | 不确定技术选型时 |
```

**原则：给执行者"去哪里找"的指针，而不是把所有东西复制过来。**

## 上下文裁剪原则

给任何人传递信息时（写 task-spec、发消息、写评审反馈），遵循以下裁剪原则：

1. **只给相关的** — 对方完成当前工作需要什么就给什么，不给"顺便了解一下"的信息
2. **给结论不给过程** — "用 Node >= 18" 而不是 "我们试过 16 不行后来升到 18 又碰到兼容问题最后..."
3. **已知的坑主动说** — 这是派活方帮执行者省时间的核心价值
4. **敏感信息脱敏** — API key 用占位符，内部路径不暴露给外部执行者
5. **宁少勿多** — 信息过载比信息不足更糟。执行者会主动追问缺失的信息，但不会主动过滤多余信息

**反面教材：**
- ❌ 把整个 brief.md 复制到 task-spec 里
- ❌ 在派活消息里复述规格书全文
- ❌ 给 CLI 工具发送项目管理系统的术语（它不懂"验收标准""用户确认点"）

## 轨迹分析方法

> 当任务失败或返工超过 3 轮时，不要只靠回忆反思。应该基于实际执行轨迹找断裂点。

### 什么时候做轨迹分析

- 任务失败（最终状态为"已放弃"或严重返工）
- 返工超过 3 轮
- 同类错误反复出现

### 轨迹分析步骤

1. **读沟通记录** — task-spec 里的沟通记录表，看卡在哪一步
2. **读状态追踪** — task-spec 状态追踪表，看状态变化是否合理
3. **读 session log**（如有）— 看 agent 实际做了什么，不只是它说做了什么
4. **找断裂点**：
   - 信号没看到（文档里有但 agent 没读/没注意）
   - 工具没用对（有更好工具但不知道/没用）
   - 规格书缺失（关键信息没写进去）
   - 验收标准模糊（理解歧义导致方向跑偏）
5. **沉淀改进**：
   - 写进 improvements.md
   - 如果是系统性问题，更新 docs/ 或 templates/

### 轨迹分析表格（填入 final-report 失败复盘）

| 步骤 | 预期行为 | 实际行为 | 断裂点 |
|------|----------|----------|--------|
| | | | _哪里出错/卡住_ |

**关键问题：**
- 卡在哪一步？
- 什么信号没看到？
- 什么工具没用对？
- 规格书缺了什么？

### 核心原则

> 改 harness，而不是一次性补锅。

- 直接修代码 = 一次性收益
- 改 harness（规则/模板/检查项）= 后续所有任务都受益

## 反馈学习闭环

项目完成后，派活方给执行者发 Feedback 消息（见 communication.md），包含"以后记住"条目。

**接收方处理规则：**
- 收到"以后记住"的执行者 → 写入自己的工作文件或经验库
- 派活方 → 更新 `resource-profiles.md` 中对应执行者的合作备忘
- 这构成跨项目的学习闭环：每次合作后双方都更聪明一点
