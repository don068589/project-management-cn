# 沟通协议

> 本文件定义的是**沟通方法论和消息标准**，不依赖任何特定平台。
> 无论通过什么工具（API 调用、消息转发、文件中转、口头传达），消息本身遵循同一套标准。
> 工具会变，标准不变。

---

## 一、核心原则

1. **消息即文档** — 每条消息结构完整，脱离对话上下文也能被理解
2. **上下文裁剪是核心能力** — 不是格式有价值，而是"帮对方整理好信息"有价值
3. **首次通信自带说明** — 对方不了解你的标准？消息里自带，无需额外培训
4. **人能读** — 无论由谁传递，都必须人也能看懂和审阅
5. **平台无关** — 不绑定任何特定工具的 API 或通信机制

## 二、沟通方式矩阵

| 执行者类型 | 特点 | 沟通注意事项 |
|------------|------|-------------|
| **有持久记忆的 agent** | 能理解上下文，可多轮对话 | 任务描述要完整，但可以追问补充 |
| **一次性执行者** | 只执行一次，没有历史 | 所有信息必须一次给齐 |
| **CLI 编码工具** | 只理解具体指令 | 需要翻译成具体编码指令（见 CLI 指令翻译） |
| **人类** | 可以反复沟通，有自主判断 | 尊重时间，一次说清楚 |

具体使用什么工具传递消息，取决于当前可用的基础设施。工具档案见 `resource-profiles.md`。

## 三、消息类型（9 种）

### 1. 派活（Ask）

```
请执行任务。规格书在 [路径]，先读规格书再执行。
执行方操作手册：[系统路径]/docs/executor.md（如不熟悉流程先读此文件）

简要说明：[一句话任务目标]
截止时间：[如有]
注意：[关键注意事项，1-2 条]
```

_对熟悉系统的执行者，可省略操作手册行。_

### 2. 催进度（Checkpoint Request）

```
任务 [TASK-ID] 状态如何？预计还需多久？
如果遇到困难请说明。
```

### 3. 返工指令（Revision）

```
任务 [TASK-ID] 评审未通过。评审记录在 [路径]。

需要修改：
1. [具体修改项]
2. [具体修改项]

修改完成后告诉我。
```

### 4. 阶段汇报（Checkpoint）

执行者主动汇报进展，不用等催。

```
任务 [TASK-ID] 进度汇报：

已完成：
1. [已完成事项]

下一步：[接下来要做什么]
风险/问题：[如有]
预计完成时间：[估算]
```

### 5. 项目交接（Briefing）

项目负责人变更或第三方介入时，打包上下文交接。

```
项目交接：[项目名]

项目状态：[一句话]
各任务进展：[逐条列出]
已做的关键决策：[决策内容及原因]
已知的坑：[需要注意的问题]
持续关注：[还没解决但需要跟进的事]
```

### 6. 知识分享（Share）

分享信息，不需要对方执行任务。

```
发现一个跟你相关的信息：
主题：[什么]
内容：[具体内容]
跟你的关系：[为什么告诉你]
```

### 7. 反馈闭环（Feedback）

项目完成后对执行者的整体反馈。

```
任务 [TASK-ID] 完成反馈：
做得好的：[具体表扬]
可以改进的：[具体建议]
以后记住：[希望执行者记住的规则/教训]
```

**"以后记住"的处理：** 接收方写入自己的经验库。构成学习闭环。

### 8. 多方转发（Relay）

需要第三方介入时，打包背景转发。

```
转发背景：
原始任务：[TASK-ID]
之前做了什么：[压缩摘要，给结论不给过程]
为什么需要你：[具体原因]
需要你做什么：[明确请求]
相关文件：[路径列表]
```

### 9. 首次通信（Introduce）

首次与不认识的执行者沟通时，消息中附带简版说明：

```
[首次合作说明]
我会用结构化的方式跟你沟通。规则很简单：
- 完成后告诉我产出位置
- 遇到问题停下来说明，不要硬做
- 写文件前检查目标位置是否已有同名文件
如果你有需要我注意的工作偏好，也请告诉我。

[正式内容]
...
```

后续通信省略此说明。

### CLI 指令翻译

CLI 工具不理解项目管理术语。翻译原则：只给"做什么"和"怎么验证"，不给项目背景。

**翻译示例：**

task-spec：`实现 GitHub Actions 自动打 tag，验收标准：push 到 main 后自动创建 vX.Y.Z tag`

翻译后：
```
在 /project 目录下创建 .github/workflows/release.yml，实现：
1. 触发条件：push 到 main 分支
2. 读取 package.json 的 version 字段
3. 用 github.rest.git.createTag 创建对应 tag
4. 测试：本地验证 YAML 语法正确
```

## 四、对话线程（Thread）

多轮沟通用 Thread 标识追踪话题：

- 首轮由发起方生成：`[项目名]-[任务简述]-[日期]`
- 后续消息沿用同一 Thread
- 轮次多时，压缩总结之前对话（给结论不给过程）
- 记录位置：task-spec 沟通记录表

## 五、沟通记录标准

```markdown
| 时间 | 方向 | Thread | 内容摘要 | 通信方式 |
|------|------|--------|----------|----------|
| 03-21 10:00 | 派活方→执行者 | release-actions-0321 | 分发 TASK-001 | [工具/渠道] |
```

记录粒度：每次发消息就记一条，摘要一句话。

## 六、公共留言板

项目目录下 `notes.md`，跨执行者的共享信息：

```markdown
| 时间 | 留言人 | 内容 |
|------|--------|------|
| 03-21 10:00 | TASK-001 执行者 | 发现 GitHub API rate limit 可能是问题 |
```

留言板用于多人需知的信息；只跟一个执行者相关的用沟通记录。

## 七、沟通方式的固化

1. 评估当前可用的通信工具和执行资源
2. 在 `resource-profiles.md` 记录每种执行者的具体工具和调用方式
3. 确定后就是标准，不用每次重新想
4. 工具变了 → 更新 resource-profiles.md，本文件不用改

## 八、怎么让执行者知道约定

| 执行者类型 | 传递方式 |
|------------|----------|
| 有持久记忆的 agent | 通过共享配置/技能文件自动加载 |
| 一次性执行者 | 在派活消息中写明（用首次通信模板） |
| CLI 工具 | 不需要约定，派活方自己解读输出 |
| 人类 | 口头或文档约定 |

**不依赖执行者"记得"约定。** 要么自动加载每次提醒，要么每次消息中写明。
