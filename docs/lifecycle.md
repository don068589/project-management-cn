# 项目生命周期管理

> 项目的暂停、取消、恢复、跨项目依赖、负责人管理、归档后处理。

---

## 项目状态

```
活跃 → 已暂停 → 恢复为活跃 / 已取消
活跃 → 已完成 → 已归档
活跃 → 已取消
```

## 暂停流程

1. 决策者指示暂停 → 记录暂停原因和当前进度
2. 通知所有进行中的执行者暂停工作
3. 更新项目索引状态为"已暂停"
4. 保留所有文档和产出，不清理
5. 恢复时：读取暂停记录，从断点继续

## 暂停恢复处理

项目暂停时有"进行中"的任务，恢复时执行者的 session 可能已不在。

**暂停时：**
1. 通知执行者保存进度和中间产出
2. 在状态追踪中记录暂停时的进度
3. 记录中间产出位置

**恢复时：**
1. 读取暂停记录
2. "进行中"的任务重新分发，补充"续做说明"：
   - 已完成的部分（不要重做）
   - 中间产出位置
   - 还需完成的部分
3. 可以分发给不同的执行者

## 取消流程

1. 决策者指示取消 → 记录取消原因
2. 通知所有执行者停止工作
3. 写简化版 final-report（已完成部分 + 取消原因）
4. 清理临时文件
5. 更新索引状态为"已取消"
6. 保留有价值的中间产出，提炼经验教训

## 项目间依赖

- brief.md 有"项目间依赖"字段，标明前置项目
- 前置未完成 → 后续可立项和拆解，但不分发执行
- 前置完成后 → 检查并启动后续项目

## 项目负责人唯一性

- 每个项目同一时间只有一个负责人（写在 brief.md）
- 只有负责人可以：更新索引、修改状态、写评审记录、分发任务
- 执行者只能：读取规格书、交付产出、反馈问题
- 换负责人 → 更新 brief.md，状态追踪中记录交接
- 不限制谁能当负责人，但同一时间只有一个

## 归档后返工

- 不重新打开已归档项目
- 新建修复项目，命名 `原项目名-fix`
- 修复项目的 brief.md 引用原项目，说明问题
- 修复完成后在原项目 final-report 补充备注
