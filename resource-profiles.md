## 一、名片管理机制

### 首次使用系统时

如果本文件全是空占位（还没有任何名片），说明你是第一次用。按以下步骤初始化：

1. **盘点可用资源** — 当前环境里有哪些执行者可以用？（agent、CLI 工具、人类）
2. **逐个探索** — 按 `philosophy.md` 元能力 1 的步骤了解每个执行者的类型、能力、边界
3. **创建名片** — 按下面的名片模板为每个执行者建档
4. **填写通信基础设施** — 在第七节记录当前可用的通信工具
5. **分发第一个任务** — 根据名片选择最匹配的执行者

**不需要一次填完所有名片。** 先填当前要用的，其他的第一次合作时再填。

### 什么时候创建名片

- **首次合作时**：第一次跟某个执行者合作，立刻创建名片
- **引入新工具时**：安装了新的 CLI 工具或接入了新平台，探索后创建名片
- **团队变动时**：新成员加入，基于初次交互创建名片

### 什么时候更新名片

- **每次项目合作后**：在合作记录中追加本次合作的摘要和教训
- **收到 Feedback 后**：把"以后记住"条目写入名片的"积累规则"
- **发现新能力/边界时**：更新能力和边界描述
- **沟通方式变化时**：更新调用方式

### 名片存在哪里

全部在本文件（resource-profiles.md）中。按类型分节管理。

### 名片的使用时机

- **分发任务前** → 读对应执行者的名片，了解能力和沟通偏好
- **首次合作前** → 先探索（见 philosophy.md 元能力 1），探索后创建名片
- **评审/反馈后** → 更新名片的合作记录和积累规则

## 六、上下文裁剪原则

给执行者信息时遵循：
1. **只给相关的** — 完成任务需要什么就给什么
2. **给结论不给过程** — "用 Node >= 18" 而不是 "我们试过 16 不行后来..."
3. **已知的坑主动说** — 省去执行者踩坑时间
4. **敏感信息脱敏** — API key 用占位符
5. **宁少勿多** — 信息过载比不足更糟，执行者会主动追问
